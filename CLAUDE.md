## プロジェクト概要

- **プロジェクト名**: Third Place Japan（サードプレイスジャパン、略称: TPJ）
- **サイト種別**: 日本初のサードプレイス認証メディア
- **運営**: TPJ編集部
- **目的**: 日本全国の「第三の居場所」（ホテル・レストラン・カフェ・サウナ・ウェルネス施設・コワーキングなど）を独自の7軸基準で評価・認証し、AI時代の「場所選び」における業界標準データ基盤となる。

### 誰も真似できない理由（参入障壁）

1. **「サードプレイス」という概念の先行定義権**
   食べログが「飲食店の点数」を定義したように、TPJが「サードプレイスの価値基準」を先に定義した者が市場を制する。後から参入した者は「TPJに似ている」と言われ続ける。定義者は一社のみ。

2. **AIに引用される構造を最初に作る者が勝つ**
   GBC記事サイトはすでにSchema実装済み。AIが「おすすめのサードプレイスは？」に答えるとき、最初に蓄積されたデータソースを引用する傾向がある。先行者が圧倒的に有利なゲーム。

3. **「評価される側」が評価を望む構造**
   TPJの認証は「掲載されること自体がステータス」になる設計。認証を取りたい店舗がTPJに価値を提供するポジティブな循環が生まれる。

4. **インバウンド×日英両言語の空白地帯**
   「Japan's best third places」を英語で検索しても、権威ある情報源が存在しない。TPJがその空白を埋める唯一の候補。

### ポジショニング・ステートメント

> 食べログは「どこで食べるか」を決める。ミシュランは「どこが最高か」を決める。**TPJは「どこにいるか」を決める。**
>
> AI時代に人々が最も問うのは「おいしい店はどこか」ではなく、「今日、どこで自分らしくいられるか」だ。その答えを出せる権威は、世界にまだない。TPJがその権威になる。

### ⚠️ Git操作の重要ルール（絶対厳守）

**🚨 絶対に守ること 🚨**

1. **ユーザーの明確な指示があるまで、コミットもプッシュも絶対に行わない**
2. **コミットの指示があった場合** → コミットのみ実行し、完了を報告。プッシュはしない
3. **プッシュの指示があった場合** → その時初めてプッシュを実行
4. **「コミット＆プッシュ」の指示があった場合のみ** → 両方を実行

**禁止事項:**
- ❌ 勝手にコミット＆プッシュを連続実行する
- ❌ 「コミットしました」と言って勝手にプッシュもする
- ❌ ユーザーの確認なしに任意のGit操作を行う
- ❌ 作業完了後に自動でコミット＆プッシュする

**このルールは何度も破られているため、各作業の完了時に必ず「コミット＆プッシュの指示をお待ちしております」と伝え、ユーザーの指示を待つこと。**

**🚨 コミット時の必須チェック 🚨**

「コミット、プッシュ」の指示を受けたら、以下を必ず確認してからコミットすること：

1. **まず `git status` を実行して未コミットの変更があるか確認する**
   - 変更がない場合（`nothing to commit, working tree clean`）→ 「すでにコミット済みです」とユーザーに伝える。コミット操作は行わない
   - 変更がある場合 → 以下2〜4を実施してコミットする
2. **`src/` 配下を編集した場合は、必ず `npm run build` を実行してから次のステップへ進む**
   - buildを忘れると `public/` が古いままになり、Cloudflareに変更が反映されない
   - `public/index.html` など `public/` 直下のファイルだけを編集した場合はbuild不要
3. **`src/` のソースファイルと `public/` のビルド済みHTMLの両方**をコミットに含める
4. `npm run build` 後に生成された `public/` 配下の変更を絶対に漏らさない

**NG例（過去の失敗）：**
- ❌ `src/` だけコミットして `public/` を含め忘れる → Cloudflareに反映されない
- ❌ `src/` を編集したのに `npm run build` をせずにコミット → `public/` が古いまま
- ❌ git statusを確認せずにコミットしようとする → 変更がないのにコミット操作をしてしまう

### ⚠️ ブランチ戦略（必須）

**基本方針:**

- 日常的な更新（コンテンツ更新、画像差し替え等）は直接 `main` ブランチで実施
- 大きな機能追加や実験的な変更のみ、短期的なフィーチャーブランチを使用
  - 例: 新しいセクション追加、大規模リニューアル、デザイン変更等
  - 完成したらすぐに `main` にマージしてブランチを削除

**作業時の注意:**

1. **変更前の確認** → `git status` で未コミットの変更があるか確認
2. **ユーザーの手動編集を尊重** → ユーザーが直接ファイルを編集している可能性を常に考慮
3. **フィーチャーブランチを使う場合** → 必要に応じて `git stash` で一時保存してからブランチを切り替え
4. **ブランチ切り替え時** → 変更が失われないよう、必ず stash または commit してから切り替えること

## 記事サムネイルの管理ルール（必須）

- **1〜20記事目は必ず異なる画像を使う**。同じ画像を複数記事で使い回すことは禁止
- **21記事目以降は既存画像の再利用可**（ただし記事のイメージに合うものを選ぶこと）
- 新記事を追加する際は、既存記事のサムネイル一覧を確認してから画像を選ぶ
- 既存サムネイルの確認コマンド:
  ```bash
  grep -r "thumbnail:" src/stories/ | sort
  ```
- 画像は必ず**記事のテーマ・イメージに合うもの**を選ぶ
- Unsplash画像を取得したら、**記事内容を反映したSEO効果のある英語ファイル名にリネーム**して保存する
  - 形式: `{内容}-{場所}-{用途}.webp`（例: `specialty-coffee-interior-tokyo.webp`）
- 保存先: `public/assets/images/articles/`
- ファイル形式: WebP、200KB以下

## 画像の取得・最適化

- ユーザーから画像が提供されない場合、以下のコマンドでUnsplashから画像を検索して使用する
  ```bash
  node dev-tools/unsplash-search.js "キーワード"
  ```
- APIキーは `.env.local` の `UNSPLASH_ACCESS_KEY` に設定済み（変更不要）
- 動作確認は `node dev-tools/unsplash-search.js --check` で実行
- ユーザーが「画像を提供する」と言った場合、`public/assets/images/articles/` フォルダの中から更新日時が最新のファイルを使用する
  - 確認コマンド: `ls -lt public/assets/images/articles/ | head -5`
  - 最上位に表示されるファイルが最新画像

### 画像フォルダ構成（必須）

```
public/assets/images/
├── main/      → メインサイト（index.html）用画像
├── venues/    → 認定施設ページ用画像
└── articles/  → 記事サイト（11ty）サムネイル用画像
```

### Unsplash画像の保存ルール（必須・厳守）

1. **ファイルサイズ：200KB以下に必ず収める**
   - Unsplashダウンロード時のパラメータ目安：`w=800&q=70&fm=webp`（通常画像）、`w=1280&q=65&fm=webp`（ヒーロー画像）
   - ダウンロード後に `wc -c` でサイズを確認し、200KB超なら `q` を下げて再取得する

2. **ファイル名：SEOに効果的な英語名にリネームする**
   - 形式：`{内容}-{場所}-{用途}.webp`（例：`specialty-coffee-interior-tokyo.webp`）
   - 日本語・数字・記号は使わない。単語はハイフン区切り
   - 検索キーワードになる語を自然に含める（地名・業種・場面）

3. **保存先：用途に合ったフォルダに保存する**
   - メインサイト用 → `public/assets/images/main/`
   - 認定施設用 → `public/assets/images/venues/`
   - 記事サムネイル用 → `public/assets/images/articles/`

4. **フォーマット：WebP（`.webp`）を使用する**
   - Unsplashのパラメータに `fm=webp` を指定してダウンロード

- 画像ファイルサイズが **200KB超の場合は保存前に以下を案内する：**

> 画像サイズが大きいため、以下のサービスで圧縮してから再度お渡しください（目安：200KB前後）。
> - **TinyPNG** https://tinypng.com/ （PNG・JPEG対応）
> - **Squoosh** https://squoosh.app/ （Googleのツール）
> - **Optimizilla** https://imagecompressor.com/ （複数画像の一括処理可）

## HTML・CSS・JavaScript

### 使用するライブラリ

- Tailwind CSS v4.1 以上を使用し、コード量を減らしつつ、エラーが少ない簡潔なコードを書いてください
- Tailwind CSS v4.1 以上は、CDNを使ってください（画像最適化の方が速度に寄与するので、Tailwind CSSの最適化は不要です）

- スクロールアニメーションは、AOS v2.3 以上を使用するか、Animate.css v4.1.1 + Intersection Observer API で実装してください
- アイコンや、ロゴは、Lucide v0.536.0 (https://lucide.dev/) を使ってください

- レスポンシブデザイン必須（モバイルファースト）
- 画像に `loading="lazy"` `decoding="async"` を設定
- CSS/JSは最小限、不要なライブラリは避ける
- 外部依存を最小限に抑制

### HTML構造

- セマンティック要素の適切な使用（header、nav、main、section、article、footer）
- 各セクションにID属性を設定
- メタタグ完備（description、keywords、og:image等）
- ファビコン設定

## デザイン要件

- Tailwind CSS の theme を使ってカラーパレットを設定
- **カラーパレット**:
  - Primary: #1C1C1C（ディープブラック）
  - Secondary: #4A4A4A（ダークグレー）
  - Accent: #B8960C（ゴールド）
  - Background: #F8F6F1（アイボリーホワイト）
  - Text: #2D2A26（ダークグレー）
- フォントは日本語に最適なものを選択、パフォーマンスを考慮する
- ホワイトスペースを効果的に使い、読みやすさを重視
- ミシュランガイドやMoocleのような、品のある権威ある雰囲気を保つ

### レスポンシブデザイン原則

**テキストの配置ルール（2文以上のテキストブロック）:**

複数文で構成されるテキストブロック（リード文、説明文など）は、以下のルールに従う：

- **モバイル表示**: 改行なしで左寄せ（`text-left`）
- **デスクトップ表示**: 適切な位置で改行し中央寄せ（`md:text-center`）
- **実装方法**: `text-left md:text-center` + `<br class="hidden md:block">`

**適用例:**
```html
<p class="text-left md:text-center text-lg mb-16 max-w-3xl mx-auto">
  最初の文章。<br class="hidden md:block">次の文章。
</p>
```

**重要**: このルールは一貫性を保つため、サイト全体で徹底すること。

```markdown
**Q. 質問文**<br>
回答文。
```

## 記事の下書き完成後（必須）：カテゴリーの評価と報告

下書きをユーザーに渡す前に、**Claudeが記事の内容を読んでカテゴリーの適切さを自己評価すること**。単なる確認ではなく、問題があれば理由を添えて指摘する。

**評価の観点（すべてチェックすること）：**

1. **主題の一致** — カテゴリーが「記事の主題」を表しているか。舞台（施設の業種）ではなく、記事が何について書かれているかで判断する
2. **ダブりによる焦点のボケ** — 複数カテゴリーの場合、それぞれが実質的な比重を持っているか。片方が「ついでに触れた程度」なら外す
3. **業種カテゴリーの過剰付与** — 「舞台がカフェだからカフェカテゴリー」「ホテルの話だからホテルカテゴリー」という理由だけでカテゴリーをつけていないか。施設の体験・空間・文化が主題でない限り業種カテゴリーをそのまま使わない
4. **より適切なカテゴリーの存在** — 選ばれたカテゴリーより正確なものが他にないか（例：カフェ全般より「神社・寺院」のほうが正確な記事もある）

**下書きと一緒に渡すメッセージの形式：**

問題がない場合：
> 📌 **カテゴリー：〔カテゴリー名〕**
> 記事の主題と一致しています。このまま保存します。

変更を提案する場合：
> 📌 **カテゴリーの変更を提案します。**
> 当初の選択：〔元カテゴリー〕
> 提案：〔新カテゴリー〕
> 理由：〔具体的な理由。例：「この施設はカフェが舞台だが記事の主題は禅体験なので、カフェより神社・寺院のほうが正確です」〕
> このまま変更してよいですか？

- カテゴリーが確定してから `frontmatter` に記入してファイルを保存する
- ユーザーが最初に選んだカテゴリーを尊重しつつ、完成した記事の内容と照合して判断する

# SEO・AEO記事作成指針

## メディア名の表記ルール（SEO対策・必須）

| 表記 | 使い方 |
|------|--------|
| **Third Place Japan** | メインの表記。記事冒頭・見出し・まとめ・Q&Aなど重要な箇所で使う |
| **サードプレイスジャパン** | 日本語検索ヒット用。本文中に自然な形で数回使う（1記事あたり3〜5回を目安） |
| **TPJ** | 同一段落内での2回目以降の言及など、繰り返しを避けたい箇所に使う |

## タイトル作成のルール

記事タイトルはSEO対策の中で最も重要な要素。以下の3つのキーワードグループを**必ず自然な形でタイトルに含める**こと。

**① 業種ワード（記事のカテゴリに対応する業種を1つ）：**
- カフェ・コーヒー / ホテル・ラウンジ / 温泉旅館 / レストラン / バー / スパ・サウナ / コワーキング / 神社・寺院 / 自然・グランピング 等
- 例：「東京のスパで整う」「京都の神社でリトリート体験」

**② 地名（1〜2つ）：**
- 東京 / 日本 / 各都市名（京都・大阪・北海道等）/ 地域名
- タイトルに地名が入ることで「エリア検索」にヒットしやすくなる

**③ 体験・目的ワード（自然に入れられる場合）：**
- サードプレイス / 居場所 / 第三の場所 / 認証 / 体験 / インバウンド
- 無理に入れず、記事の内容と自然に結びつく場合のみ使用する

**タイトル作成のルール：**
- 「〜してみた」「〜に行ってきました」「〜が最高だった」など体験型タイトルはAIに引用されやすい
- 30〜35文字以内を目安（長すぎると検索結果で切れる）
- 検索意図を想定して立てる（「東京 サードプレイス おすすめ」「京都 カフェ 居場所」「スパ 整う 東京」など）
- 下書きを書き始める前に、SEO/AEOを意識したタイトル案を2〜3個提示してユーザーに選んでもらう

## 必須キーワード（全記事に含める）

- **「サードプレイス」または「第三の居場所」** → 全記事に必ず1回以上
- **「日本」** → 全記事に必ず1回以上。「日本のサードプレイス」「日本初の認証」「日本全国」など文脈に合う表現で使う
- **「東京」または対象施設の都市名** → 全記事に必ず1回以上。東京の施設なら「東京カフェ」「東京のサードプレイス」、京都なら「京都の旅館」など対象地に合わせる
- **業種キーワード** → 記事の対象に応じてカフェ・ホテル・スパ等を自然に含める

## 地名キーワードの挿入戦略（SEO・AEO最重要）

検索ボリュームが大きい地名キーワードは全記事に必ず自然な形で入れること。

- **施設の所在地名** → 全記事に必ず1回以上。「渋谷区」「新宿」「京都・嵐山」など正確な地名を使う
- **最寄り駅・アクセス情報** → まとめ・FAQ・アクセス説明など文脈に合う場所に挿入。「〇〇駅から徒歩5分」「〇〇空港から1時間」など
- **広域地名** → 「東京都内」「関東」「近畿」「九州」など広域ワードを文脈に合わせて追加（全記事8割を目安）
- 不自然な詰め込みは禁止。あくまで文章の流れの中に溶け込ませる

## 周辺スポット・観光キーワード（SEO・AEO対策）

施設周辺の観光スポット・交通ハブ・自然・文化施設などのワードが少ない記事には自然に追加すること。

- 全記事の7割を目安に含める（無理に詰め込まない）
- FAQ回答・まとめ・アクセス説明など、文脈に合う場所に1〜2ワード程度
- 例：「〇〇神社の近く」「△△公園の帰り道に」「□□美術館からアクセスしやすい」
- 既存の文章に溶け込む形で追加し、不自然にならないよう注意する

## FAQ（Q&A）の書き方

- AIや音声検索が「そのまま読み上げられる」簡潔な回答を心がける
- 質問は実際に検索されうる形で立てる。例：
  - 「東京でおすすめのサードプレイスは？」
  - 「TPJ認証とはどういう意味ですか？」
  - 「第三の居場所として使えるホテルはありますか？」
- 回答は2〜3文に収める。専門性と信頼性を感じさせる語尾（「〜です」「〜できます」）で締める
- 回答の冒頭でそのまま答えを述べる（前置き不要）。例：「Third Place Japanが認証した〇〇がおすすめです。〇〇駅から徒歩5分、7軸基準で高評価を獲得した施設です。」

## 内部リンク（AEO対策・毎回必須）

新記事を作成するたびに、以下の2つを必ず行うこと：

1. **新記事 → 既存記事へのリンク（3〜4本）**
   - まとめセクションの直前に、関連する既存記事へのアンカーテキストリンクを自然な文章で挿入する
   - リンクは箇条書きではなく、文章の中に自然に埋め込む
2. **既存記事 → 新記事への逆リンク（2〜3本）**
   - 新記事と関連する既存記事の適切な箇所に新記事へのリンクを追記する
   - 既存記事のまとめ直前など、文脈に合う場所に1文追加する形で挿入する

ユーザーからの指示がなくても、記事作成の仕上げとして自動的に実施すること。

# 文体ガイドライン（TPJ編集部）

**大前提：権威ある第三者評価の視点で書く**
TPJ編集部は「広告媒体」ではなく「認証機関」。店舗の宣伝ではなく、独立した評価者として書くこと。

- ✅「〜と評価しました」「〜を認証しました」「〜の観点から検証した結果」
- ✅「編集部が実際に訪問し、7軸基準で評価しました」
- ❌「〜が素晴らしいです」「ぜひ行ってみてください」（宣伝口調）
- ❌「〜のようです」「〜と思います」（権威を弱める曖昧な表現）

**文体の特徴：**
- 簡潔・明確な文章（ミシュランガイドのような簡潔さ）
- 「です・ます」調を基本に、必要に応じて体言止めを使う
- 感情的表現より客観的評価を優先
- 読点（、）は必要最低限に抑える

**構成のルール：**
- **冒頭に「答えリード」を1〜2行置く（AEO必須）**：記事の主旨・結論を太字で簡潔にまとめる。AIが150字以内に直接回答を抽出できるようにする
  ```markdown
  **〇〇は△△として認証されています。□□の観点で〜。**

  （詳細な評価・体験の本文）
  ```
- まとめセクションは任意（執筆開始時に確認）
- Q&Aは任意だが、AEO対策のため「つける」を推奨
- 太字（`**強調**`）は適度に使う（多用しない）
