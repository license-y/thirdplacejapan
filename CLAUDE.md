- **プロジェクト名**: Third Place Japan（サードプレイスジャパン、略称: TPJ）
- **サイト種別**: 日本初のサードプレイス認証メディア
- **運営**: TPJ編集部

1. **「サードプレイス」という概念の先行定義権**

2. **AIに引用される構造を最初に作る者が勝つ**
   GBC記事サイトはすでにSchema実装済み。AIが「おすすめのサードプレイスは？」に答えるとき、最初に蓄積されたデータソースを引用する傾向がある。先行者が圧倒的に有利なゲーム。

3. **「評価される側」が評価を望む構造**
   TPJの認証は「掲載されること自体がステータス」になる設計。認証を取りたい店舗がTPJに価値を提供するポジティブな循環が生まれる。

4. **インバウンド×日英両言語の空白地帯**
   「Japan's best third places」を英語で検索しても、権威ある情報源が存在しない。TPJがその空白を埋める唯一の候補。

> 食べログは「どこで食べるか」を決める。ミシュランは「どこが最高か」を決める。**TPJは「どこにいるか」を決める。**
>

### ⚠️ Git操作の重要ルール（絶対厳守）

**🚨 絶対に守ること 🚨**

1. **ユーザーの明確な指示があるまで、コミットもプッシュも絶対に行わない**
2. **コミットの指示があった場合** → コミットのみ実行し、完了を報告。プッシュはしない
3. **プッシュの指示があった場合** → その時初めてプッシュを実行
4. **「コミット＆プッシュ」の指示があった場合のみ** → 両方を実行

**禁止事項:**
- ❌ 勝手にコミット＆プッシュを連続実行する
- ❌ 「コミットしました」と言って勝手にプッシュもする
- ❌ ユーザーの確認なしに任意のGit操作を行う
- ❌ 作業完了後に自動でコミット＆プッシュする

**このルールは何度も破られているため、各作業の完了時に必ず「コミット＆プッシュの指示をお待ちしております」と伝え、ユーザーの指示を待つこと。**

**🚨 コミット前の確認（絶対厳守・新ルール） 🚨**

「コミット、プッシュ」の指示を受けても、**すぐに実行してはいけない**。
必ず以下の順序で確認してからユーザーの承認を得ること：

1. `git status` を実行して変更ファイル一覧をユーザーに見せる
2. 「以下のファイルをコミット・プッシュします。よろしいですか？」と確認する
3. ユーザーが「はい」「OK」「進めて」などの承認をしてから実行する

❌ 確認なしに即実行するのは禁止（過去に予期しないファイルを含めてコミットした失敗あり）

---

**🚨 コミット時の必須チェック 🚨**

承認を得たら、以下を実施してコミットすること：

1. **まず `git status` を実行して未コミットの変更があるか確認する**
   - 変更がない場合（`nothing to commit, working tree clean`）→ 「すでにコミット済みです」とユーザーに伝える。コミット操作は行わない
   - 変更がある場合 → 以下2〜4を実施してコミットする
2. **`src/` 配下を編集した場合は、必ず `npm run build` を実行してから次のステップへ進む**
   - buildを忘れると `public/` が古いままになり、Cloudflareに変更が反映されない
   - `public/index.html` など `public/` 直下のファイルだけを編集した場合はbuild不要
   - **`src/_data/` 配下（venues.json・areas.json等）を編集した場合も必ず `npm run build` が必要**
     データファイルはビルド時にHTMLへ展開される。データだけコミットしても画面には反映されない。
3. **`src/` のソースファイルと `public/` のビルド済みHTMLの両方**をコミットに含める
4. `npm run build` 後に生成された `public/` 配下の変更を絶対に漏らさない
5. **`public/pagefind/` も必ずコミットに含める**（検索インデックス。含めないと検索窓が表示されなくなる）

**NG例（過去の失敗）：**
- ❌ `src/` だけコミットして `public/` を含め忘れる → Cloudflareに反映されない
- ❌ `src/` を編集したのに `npm run build` をせずにコミット → `public/` が古いまま
- ❌ `src/_data/` のJSONを更新したのに `npm run build` をせずにコミット → 画面に反映されない
- ❌ git statusを確認せずにコミットしようとする → 変更がないのにコミット操作をしてしまう
- ❌ `public/pagefind/` をコミットし忘れる → 検索窓がサイト上で消える（2026-07-01に発生した障害）

**📌 `git status` で大量のファイルが変更されている場合について（正常な動作）：**

レイアウト（`src/_layouts/`）やインクルード（`src/_includes/`）を変更すると、
`npm run build` がそれらを使用するすべてのHTMLを再生成するため、
`public/` 配下の数百ファイルが変更済みとして表示される。
これは意図通りの正常な動作であり、すべてコミットに含めてよい。

- `src/_includes/nav-stories.njk`（共通ヘッダー）を変更 → 全記事・全エリアページが再生成される
- `src/_layouts/article.njk`（記事レイアウト）を変更 → 全記事ページが再生成される
- `src/_data/areas.json` を変更 → 全エリアページが再生成される

### ⚠️ ブランチ戦略（必須）

**基本方針:**

- 日常的な更新（コンテンツ更新、画像差し替え等）は直接 `main` ブランチで実施
- 大きな機能追加や実験的な変更のみ、短期的なフィーチャーブランチを使用
  - 例: 新しいセクション追加、大規模リニューアル、デザイン変更等
  - 完成したらすぐに `main` にマージしてブランチを削除

**作業時の注意:**

1. **変更前の確認** → `git status` で未コミットの変更があるか確認
2. **ユーザーの手動編集を尊重** → ユーザーが直接ファイルを編集している可能性を常に考慮
3. **フィーチャーブランチを使う場合** → 必要に応じて `git stash` で一時保存してからブランチを切り替え
4. **ブランチ切り替え時** → 変更が失われないよう、必ず stash または commit してから切り替えること

**例**:
```

## 画像の取得・最適化

- ユーザーから画像が提供されない場合、以下のコマンドでUnsplashから画像を検索して使用する
  ```bash
  node dev-tools/unsplash-search.js "キーワード"
  ```
- APIキーは `.env.local` の `UNSPLASH_ACCESS_KEY` に設定済み（変更不要）
- 動作確認は `node dev-tools/unsplash-search.js --check` で実行
- ユーザーが「画像を提供する」と言った場合、`public/assets/images/articles/` フォルダの中から更新日時が最新のファイルを使用する
  - 確認コマンド: `ls -lt public/assets/images/articles/ | head -5`
  - 最上位に表示されるファイルが最新画像

### 画像フォルダ構成（必須）

```
public/assets/images/
├── main/      → メインサイト（index.html）用画像
├── venues/    → 認定施設ページ用画像
└── articles/  → 記事サイト（11ty）サムネイル用画像
```

### Unsplash画像の保存ルール（必須・厳守）

1. **ファイルサイズ：200KB以下に必ず収める**
   - Unsplashダウンロード時のパラメータ目安：`w=800&q=70&fm=webp`（通常画像）、`w=1280&q=65&fm=webp`（ヒーロー画像）
   - ダウンロード後に `wc -c` でサイズを確認し、200KB超なら `q` を下げて再取得する

2. **ファイル名：SEOに効果的な英語名にリネームする**
   - 形式：`{内容}-{場所}-{用途}.webp`（例：`specialty-coffee-interior-tokyo.webp`）
   - 日本語・数字・記号は使わない。単語はハイフン区切り
   - 検索キーワードになる語を自然に含める（地名・業種・場面）

3. **保存先：用途に合ったフォルダに保存する**
   - メインサイト用 → `public/assets/images/main/`
   - 認定施設用 → `public/assets/images/venues/`
   - 記事サムネイル用 → `public/assets/images/articles/`

4. **フォーマット：WebP（`.webp`）を使用する**
   - Unsplashのパラメータに `fm=webp` を指定してダウンロード

- 画像ファイルサイズが **200KB超の場合は保存前に以下を案内する：**

> 画像サイズが大きいため、以下のサービスで圧縮してから再度お渡しください（目安：200KB前後）。
> - **TinyPNG** https://tinypng.com/ （PNG・JPEG対応）
> - **Squoosh** https://squoosh.app/ （Googleのツール）
> - **Optimizilla** https://imagecompressor.com/ （複数画像の一括処理可）

### レスポンシブデザイン原則

**テキストの配置ルール（2文以上のテキストブロック）:**

複数文で構成されるテキストブロック（リード文、説明文など）は、以下のルールに従う：

- **モバイル表示**: 改行なしで左寄せ（`text-left`）
- **デスクトップ表示**: 適切な位置で改行し中央寄せ（`md:text-center`）
- **実装方法**: `text-left md:text-center` + `<br class="hidden md:block">`

**適用例:**
```html
<p class="text-left md:text-center text-lg mb-16 max-w-3xl mx-auto">
  最初の文章。<br class="hidden md:block">次の文章。
</p>
```

**重要**: このルールは一貫性を保つため、サイト全体で徹底すること。

**例外（2026-07-09追加）**: リンクテキストなど「途中で割れると意味が壊れる短い語句」（例：「TPJについて知る」）の直前に置く `<br>` は、`hidden md:block` にせず常時表示にする。モバイル幅で語句の途中（例：「TPJ」と「について知る」の間）に自動改行が入り、読みにくくなるのを防ぐため。対象例：`src/_layouts/venue.njk` の底部リンク行。

# AEO対策 重要事項

## JSON-LD構造化データ（必須）

以下の構造化データを `<head>` 内または `</body>` 直前に設置:

1. **Organization**: TPJ組織情報
2. **FAQPage**: よくある質問
3. **WebSite**: サイト情報
4. **ItemList / Review**: 認証場所の一覧・評価データ（認証ページ）

## FAQセクション

- アコーディオン形式で実装
- AIや音声検索が「そのまま読み上げられる」簡潔な回答を心がける
- JSON-LD FAQPage と連動

## 記事FAQ（よくある質問）の書式ルール

Markdownでは1行の改行が無視されるため、**Q.**タイトルと回答が同じ行に連結して表示される。必ず `<br>` を使って明示的に改行すること。

```markdown
**Q. 質問文**<br>
回答文。
```

## コンテンツの書き方

- 自然で人間味のある文章（AIが引用しやすい）
- 具体的な情報を明記（場所名、住所、評価軸スコアなど）
- 「サードプレイス」「第三の居場所」「居場所」などのキーワードを自然に含める
- 業種横断キーワード（カフェ・ホテル・スパ・コワーキング・サウナ）を文脈に合わせて使う

**旧ルール（2026-07-05）では「グレード表記禁止」としていたが、2026-07-06のグレード機能実装により方針変更。グレードバッジは施設ページ・一覧ページの両方で表示する。**

## 「実証店舗」と「TPJ認証」の関係（用語定義・2026-07-07明文化）

サイト内でAIに矛盾なく引用されるよう、両者の関係をここで一度明文化する。

- **TPJ認証**：Certified → Silver → Gold → Platinum → Flagshipの5段階からなる、施設の申請・審査を経て発行される正式な掲載制度そのもの
- **実証店舗**：TPJ認証の中でも**Flagship施設のうち、TPJ自身が空間設計の知見を継続的に検証する特別な位置づけ**を指す呼称。TPJ認証とは別の独立した概念ではなく、**Flagshipのサブセット**
- **関係式**：実証店舗 ⊂ Flagship ⊂ TPJ認証（すべての実証店舗はFlagshipだが、すべてのFlagshipが実証店舗になるわけではない。現在は Green Beans Coffee のみが実証店舗）
- 記事・FAQ・コピーを書く際は、この包含関係を崩さないこと。「実証店舗＝TPJ認証と同じもの」と誤読される書き方（例：両者を並列に並べて対比させる文）は避ける

## Flagship施設の特別表示ルール（2026-07-05追加・2026-07-06更新・2026-07-09パンくず改訂）

- 説明文：「Third Place Japanの実証店舗として、理想の空間づくりを検証しています。」

## venues.json 施設データ管理ルール

- 施設データは `src/_data/venues.json` に一元管理する
- `published: true` の施設のみが一覧・詳細ページに表示される（非表示にする場合は `published: false`）
- `grade_slug: "flagship"` の施設は一覧ページの「Flagship」固定枠に表示される（現在はGBC 1件）
- `grade_slug` が flagship 以外（certified / silver / gold / platinum）の施設が「TPJセレクト」グリッドに表示される
- **施設を追加する際は必ず `project-docs/TPJセレクト掲載基準ルールブック.md` の7基準・カテゴリ配分・大手禁止ルールを確認すること**
- **一覧ページ（`src/stories/select/index.njk`）のTPJセレクトグリッドは新着順（新しい施設が上部）に表示する**（2026-07-07確定）。実装は `{% for v in published | reverse %}`。`venues.json` には末尾に追記していくため、`reverse` で新着が先頭に来る

## TPJセレクト 掲載仕様（2026-07-06確定・2026-07-07更新）

TPJセレクトは、認証グレードとは別軸の「編集部招待制」枠。施設からのお申し込み不要。
**参照ページ（原則ルール）**：https://thirdplacejapan.com/stories/select/onibus-coffee-nakameguro/

| 項目 | 仕様 |
|---|---|
| `grade` | `"TPJセレクト"` |
| `grade_slug` | `"select"`（certified/silver/gold/platinum/flagship のいずれでもない値） |
| citation | 100〜150字。編集部視点の一言紹介 |
| FAQ | 2〜3問。「どんな場所か」「アクセス」など基礎情報のみ。**営業時間・価格帯は不確定要素のためNG** |
| FAQ順序 | 自然な流れで配置する。例：①どんな場所か → ②施設の特徴 → ③アクセス |
| tags | **なし（設定しない）** |
| 表示ボタン | **「関連サイト」「地図を見る」のみ**。Instagram・電話・メニュー等は表示しない |
| 営業時間・価格帯表示 | **非表示**（上部バー・下部asideともに。`grade_slug == "select"` で自動制御） |
| 地図確認コメント | 「地図を見る」ボタン下部に自動表示：「営業日・営業時間・電話番号など店舗の詳細情報は変動することがあります。「地図を見る」から最新の店舗情報をご確認ください。」 |
| 底部テキスト | 「この場所は Third Place Japan セレクト店舗です」＋「TPJについて知る」「グレードとは？」の2リンク（グレー表示） |
| 写真 | 専用写真なし（デフォルト画像 `/assets/images/main/japan-cafe-interior-hero.webp` のまま） |
| JSON-LD | geo・sameAs・openingHours 等のグレード別拡張プロパティは出力されない（thisLevel=0） |

**バッジ・ロゴの表示ルール（TPJセレクト専用）：**
- **5段階認証グレード（Flagship / Platinum / Gold / Silver / Certified）のバッジ・ロゴは一切表示しない**
- 表示するのは「TPJセレクト」バッジ（虫眼鏡アイコン）のみ
- グレード凡例バー・グレード関連の視覚要素もTPJセレクト施設ページには表示しない
- `grade_slug: "select"` にすることで template が自動制御する

**認証グレードとの違い：**
- TPJセレクトは「編集部の目利き」であり、商取引・申請・審査は発生しない
- タグ・FAQ は最小限。情報の厚みは認証グレードに委ねる設計

## TPJセレクト 候補施設の探し方（Claude検索フロー）

ユーザーから「〔エリア〕の〔カテゴリ〕でTPJセレクト候補を探してください」と指示されたら、以下のフローで対応する。

1. **Web検索**で対象エリア×カテゴリの施設を調査する
2. **TPJ 7軸基準**（居心地・静けさ・特別感・ストーリー・再訪価値・記録性・インバウンド）と**非大手・独立系・居場所としての価値**で評価する
3. 候補を**2〜3件**提示する（各施設に選定理由を添える）
4. ユーザーが選んだ施設について「**Googleマップの共有URLを貼ってください**」と案内する
5. 受け取ったURLから `maps_url` と `lat`/`lng` を取得し、venues.json に追加する

**指示の例：**
- 「恵比寿のカフェでTPJセレクト候補を探してください」
- 「東京のコワーキングスペースでTPJセレクト候補を3件探してください」

**追加時の自動処理：**
- `grade: "TPJセレクト"` / `grade_slug: "select"` で設定
- citation（100〜150字）を生成
- FAQ 2〜3問を生成（営業時間NG・自然な順番）
- タグなし・写真はデフォルト（`/assets/images/main/japan-cafe-interior-hero.webp`）

## citation（紹介文）バリエーション化ルール

- **「Third Place Japan編集部がTPJセレクトに選出。」という完全に同一の一文を複数施設で使い回すことは禁止**
- 施設ごとに表現を変えた文章を書く。最低5パターン程度を意識してローテーションする
  - 例：「〜として、静かに紹介します。」「〜と出会いました。」「〜をTPJセレクトに加えます。」「〜が選ばれた理由はここにあります。」
- 新規施設を追加する際は、既存citationの文体を参照して重複しないパターンを選ぶ
- **理由**：件数が増えるほど機械的な量産コンテンツに見え、SEO上「重複表現の多用」として不利になる可能性がある

**「サードプレイス」「third place」併記ルール（2026-07-21追加）**：citationには「サードプレイス」（カタカナ）と英語表記「third place」の両方を自然な形で含めること。理由：TPJセレクトのcitationは`description: "{{ venue.citation }}"`としてそのままmeta descriptionに直結するため、ここに検索語が入っていないと施設個別ページ（最もロングテールで検索・AI引用されやすいページ群）が肝心のキーワードを一切拾えない。日本語検索（サードプレイス）と英語圏・AIが処理する国際的な概念語（third place）の両方をカバーする狙いもある。2026-07-21の監査で公開施設23件中19件が、citation・FAQ・title・meta descriptionのいずれにもこのキーワードを一切含んでいないことが判明し、全件是正済み。上記のバリエーション化ルール（同一フレーズの使い回し禁止）は引き続き適用する。

## 施設情報カード 設備バッジルール（2026-07-09追加）

施設詳細ページ（`venue.njk`）の施設情報カード（営業時間・価格帯・電話の下）に、設備・特徴バッジを自動表示する。**「メニューを見る」リンクはこのカードから廃止済み**（下部asideの「メニュー」ボタンは維持）。

### 表示条件（venues.jsonの既存フィールドから自動判定・データ駆動）

| バッジ | 表示条件 | アイコン（Lucide） |
|---|---|---|
| 自家焙煎コーヒー専門店 | `tags` に「自家焙煎機」を含む | `coffee` |
| ○○種類のコーヒー豆 | `bean_varieties_count` が設定されている | `bean` |
| Wi-Fi・電源完備 | `amenity_features` に「Wi-Fi」と「電源」を両方含む | `wifi` |
| グルテンフリー対応 | `tags` に「グルテンフリー」を含む | `wheat-off` |
| 店内ペット同伴可 | `tags` に「店内ペット同伴可」を含む | `paw-print` |
| 禁煙 | `smoking_allowed` が `false` | `cigarette-off` |

### レイアウト仕様
- 該当バッジが1つでもあれば、営業時間・価格帯・電話の行の下に区切り線（`border-t`）を挟んで表示する
- `grid grid-cols-3 gap-2` で3列グリッド固定。バッジ数に応じて自動的に複数行になる（6個なら2段×3列）
- アイコンは `<i data-lucide="アイコン名">` 形式で埋め込む（`lucide.createIcons()` は `base.njk` で実行済みのため自動レンダリングされる）。venue.njk内の他アイコン（営業時間・価格帯・電話）は手書きSVGだが、このバッジ行のみLucideタグ方式を使う
- 各バッジのスタイルは `flex items-center justify-center gap-1.5 text-xs px-3 py-2 rounded-full bg-white border border-gold/50` を踏襲する（枠線は2026-07-09に`border-stone-200`から金枠`border-gold/50`へ変更済み。視認性向上のため）

### bean_varieties_count フィールド（新規・任意フィールド）
- コーヒー豆の取り扱い品種数を訴求したい施設にのみ設定する（数値）
- 例：`"bean_varieties_count": 30` → 「30種類のコーヒー豆」バッジを表示
- 現在の設定施設：Green Beans Coffee（30種類）

### 新しいバッジを追加する場合の注意（必須）
- 特定施設名をテンプレートにハードコードせず、`venues.json` の既存/新規フィールドから自動判定するデータ駆動方式にする
- グリッドは3列固定。バッジ総数が3の倍数でなくても `grid` により自動折り返しされる（最終行は左詰め）
- 新規アイコンを使う場合はLucide v0.536.0に実在するか事前確認する（`https://unpkg.com/lucide-static@0.536.0/icons/{アイコン名}.svg` が200を返すか確認してから使用する）

# TPJ記事管理（11ty / /stories/）

## 概要

11ty（Eleventy）を使った記事管理システム。`/stories/` 以下で動作。

## ビルドコマンド

```bash
npm run dev    # ローカル確認（検索機能は動作しない）
npm run build  # 本番ビルド（Cloudflare Pagesが自動実行）
```

**重要**: `npm run dev` では検索（Pagefind）は動作しない。検索テストは `npm run build` 後に `npx serve public` で確認。

## 検索UIの設定値

- **1回の表示件数（PAGE_SIZE）：25件**（`src/_includes/nav-stories.njk` 内の `const PAGE_SIZE = 25`）
  - 初回表示・「さらに表示」ボタン押下ともに25件ずつ表示する
  - 変更する場合はこの定数のみ編集すればよい

## フッターの「最新記事」・「記事一覧」ルール（2026-07-05確定）

### 記事一覧リンクのURL

- **日本語フッター**（`src/_includes/footer-stories.njk`）の「記事一覧 →」は `/stories/latest/` を指す
- **英語フッター**（`src/_includes/footer-journal.njk`）の「All Articles →」は `/en/stories/` を指す
  （`/en/stories/latest/` は未作成のため。作成した場合は `/en/stories/latest/` に変更する）

### 最新記事の取得方法

`collections.articles` はすでに **新しい順（date降順）** でソート済み（`.eleventy.js` の `addCollection` で定義）。

```njk
{# 正しい実装 #}
{% set latestPosts = collections.articles | rejectLang("en") | head(4) %}

{# ❌ 禁止：| reverse を使うと古い記事4件が取れてしまう #}
{% set latestPosts = collections.articles | reverse | head(4) %}
```

- `| reverse` は使わない
- 日本語フッターでは `rejectLang("en")` で英語記事を除外する
- 英語フッターでは `head(4)` のみ（英語記事フィルタリングは今後の課題）

### TPJガイド記事の取得

- `collections.aboutArticles`（`category_slug: "about"` かつ `tags: articles` の記事を新しい順で返す）を使う
- 一覧リンクは `/stories/about/` を指す

### 「TPJガイド」ピラー記事ルール（2026-07-20確定・`src/stories/about/index.njk`）

`/stories/about/` はナビ・フッター・パンくず（他ページ側）で以前から「TPJガイド」と呼ばれていたが、ページ自身は「サードプレイスとは」表記のままで不一致だった。2026-07-20に**ページ自体をピラー記事として全面改稿**し、名称を統一した。

**確定した構造：**
1. **本文（ピラー記事）**：2,800〜4,000字を目安に、結論先出し・8セクション構成（定義／8条件／TPJ7軸／3つの場所の違い／「居場所」との使い分け／具体例／理論の源流／TPJの認証を知る）＋FAQ（5問）＋まとめ。CLAUDE.md「ピラー記事生成プロンプト」の構成規定に準拠
2. **各セクション本文中に、対応する既存`about`記事へインラインリンク**（ハブ＆スポーク構造。詳細は個別記事に委ね、本文は要約に留める）
3. **本文の下に「さらに詳しく知る」として、既存12記事を4テーマ（サードプレイスの基礎／8条件を深く読む／使い分けと実例／理論の源流）＋認証グレードへのカードリンクで再掲**（従来は時系列の単純リストのみだった）
4. **H1・パンくずを「TPJガイド」に統一**（旧「サードプレイスとは」から変更。ナビ・フッター・他ページのパンくず表記とようやく一致）
5. **`/stories/about/certification-grades/` への導線を新設**（本文セクション⑧「TPJの認証を知る」＋カードリンク。従来この認証グレード説明ページへは`/stories/about/`から一切リンクされておらず孤立していた）

**テーマ別グルーピングの実装方法（`index.njk`）：**
```njk
{% set group1 = ["what-is-third-place", "third-place-meaning", ...] %}
{% macro renderGroup(title, slugs, posts) %}
  {% for post in posts %}
    {% if post.fileSlug in slugs %}
      {# カード表示 #}
    {% endif %}
  {% endfor %}
{% endmacro %}
```
`post.fileSlug`（Eleventyの標準プロパティ）でグループ分けする。`selectattr`は使わず、`in`演算子での手動フィルタリングを使う（既存の`rejectLang`/`selectLang`カスタムフィルターと同様、Nunjucksの`selectattr`は複雑な条件で不具合が出やすいため）。

**今後、`about`カテゴリに新しい記事を追加する場合：**
- `collections.aboutArticles`には自動的に含まれるが、`src/stories/about/index.njk`本文中のテーマグループ（`group1`〜`group4`）・本文セクションへの言及・「さらに詳しく知る」への追加は**手動で行う必要がある**（自動反映されない）
- 新記事がどのテーマに属するかを判断し、該当する`renderGroup`呼び出しの配列に`fileSlug`を追加すること

## フロントマターのテンプレート

```markdown
---
layout: article.njk
title: 記事タイトル
description: 110〜120字。一文目で結論を完結させ、AI単体引用に耐える密度で書く
date: 2026-06-14
tags:
  - articles
  - カテゴリ名
category_slug: カテゴリslug
area_slug: エリアslug（任意・エリアページに記事を表示する場合）
area_name: エリア名（area_slugを設定した場合は必須）
thumbnail: /assets/images/articles/xxx.jpg
---
```

**📌 `description` の文字数ルール：日本語は110〜120字、英語は150〜160字（2026-07-20改訂・言語別に分離）**

**日本語記事（`src/stories/`）：110〜120字**
以前は「115〜160字」としていたが、**120字＝検索結果・SNSカード等でdescriptionが途切れずに全文表示される目安（特にPC表示）**であることから、110〜120字に改訂した（2026-07-19）。160字まで書くと表示上は末尾が「…」で切れてしまい、一文目で結論を完結させても後半が見えないケースが生じるため、全文が途切れず見える範囲に収める。
- 下限110字：AI単体引用に耐える情報密度を保つための最低ライン
- 上限120字：PC検索結果・OGPカード等で切れずに表示される目安
- 110〜120字に収まらない場合は、一文目（結論文）を短くする・二文目以降を削るなどして調整する

**英語記事（`src/en/stories/`）：150〜160字（2026-07-20新設）**
日本語の110〜120字ルールを英語記事にもそのまま適用していたが、2026-07-20の監査でEN22本全件が147〜220字と実態が乖離していることが判明し、英語専用の基準を新設した。英語のmeta descriptionはGoogleのSERP上で概ね155〜160文字前後（約920px）まで表示されるのが一般的な目安で、全角文字を前提にした日本語の120字とは表示幅の前提が異なるため、120字へ単純に削ると情報密度が落ちすぎる。
- 下限150字：AI単体引用に耐える情報密度を保つための最低ライン（日本語版と同じ考え方）
- 上限160字：Google SERPで途切れずに表示される一般的な上限の目安
- 150〜160字に収まらない場合は、地名・固有名詞は残しつつ形容表現（“a single”“three distinct”等の冗長な修飾）を削って調整する

**🚨 `category_slug` は必須フィールド（絶対に省略しない）🚨**

- カテゴリページ（`/stories/{slug}/`）への記事表示は `category_slug` で制御される
- `tags` にカテゴリ名を入れるだけでは**カテゴリページに表示されない**
- カテゴリslugの一覧は `src/_data/categories.json` を正とする
  - 例：カフェ・スペシャルティコーヒー → `cafe-specialty-coffee`
  - 例：コワーキング・シェアオフィス → `coworking`
  - 例：神社・寺院・パワースポット → `shrine-temple`
- 記事生成時・ファイル保存前に必ず確認すること

**`area_slug` / `area_name` のルール**

- エリアページ（`/stories/area/{pref}/{city}/{area}/`）に記事を表示したい場合に設定する
- `area_slug` は `src/_data/areas.json` に定義されたスラッグを使う（例：`omotesando`、`shibuya`、`ginza`）
- `area_slug` を設定する場合は `area_name` も必ずセットで設定する（例：`area_name: 表参道`）
- 記事の**主題がそのエリアについて書かれている**場合のみ設定する。施設が所在するだけの場合は設定しない
- 1記事につき1エリアのみ設定可（複数エリアにまたがる記事は最もメインのエリアを選ぶ）
- 設定を忘れると該当エリアページに記事が表示されないため、記事保存前に必ず確認すること
- **既存記事にも遡って設定すること**。`area_slug` は新機能として導入されたため、過去に作成した記事には設定されていない。エリアについて書かれた既存記事を見つけたら、都度 `area_slug` / `area_name` を追加してビルド・コミットすること

**🚨 `area_name` は全記事に必須（新ルール）🚨**

- **`area_name` はすべての記事に設定すること**。`/stories/` トップの「最新記事」セクションでエリア名を表示するために使用する
- 特定の街・エリアを主題とする記事 → そのエリア名を設定（例：`area_name: 銀座`）
- 複数エリアにまたがる区レベルのピラー記事 → 対象の区名を設定（例：`area_name: 中央区`、`area_name: 渋谷区`）
- `area_name` のみ設定し `area_slug` を省略することも可（表示のみ目的の場合）
- `area_slug` は引き続き任意。エリアページへの記事掲載が必要な場合のみ設定する

**🚨 記事ページのエリア階層リンク表示ルール 🚨**

記事ページ（`src/_layouts/article.njk`）の日付・著者行の右側に、エリア階層リンクが自動表示される。

| フロントマター設定 | 表示内容 |
|---|---|
| `area_slug` あり | `東京都 / 中央区 / 銀座`（3階層・各リンク付き） |
| `area_name` のみ（区名）| `東京都 / 中央区`（2階層・各リンク付き） |
| `area_name` のみ（マッチなし）| `銀座`（テキストのみ、リンクなし） |
| どちらも未設定 | 非表示 |

**実装フィルター（`.eleventy.js`）：**
- `areaInfo(areaSlug)` — `area_slug` → 都道府県・市区町村・エリアの情報オブジェクトを返す
- `cityInfo(cityName)` — `area_name`（区名）→ 都道府県・市区町村の情報オブジェクトを返す
- データソース：`src/_data/areas.json`

**注意：**
- `about` カテゴリの記事はエリア表示対象外（日付・著者行自体が非表示）
- `area_name` を区名（例：`中央区`）に設定すると `cityInfo` が都道府県まで補完する
- `area_name` をエリア名（例：`銀座`）に設定した場合、`cityInfo` で都市がマッチしないためテキストのみ表示（リンクなし）。エリアページへのリンクが必要なら `area_slug` も設定すること

## 記事の下書き完成後（必須）：カテゴリーの評価と報告

下書きをユーザーに渡す前に、**Claudeが記事の内容を読んでカテゴリーの適切さを自己評価すること**。単なる確認ではなく、問題があれば理由を添えて指摘する。

**評価の観点（すべてチェックすること）：**

1. **主題の一致** — カテゴリーが「記事の主題」を表しているか。舞台（施設の業種）ではなく、記事が何について書かれているかで判断する
2. **ダブりによる焦点のボケ** — 複数カテゴリーの場合、それぞれが実質的な比重を持っているか。片方が「ついでに触れた程度」なら外す
3. **業種カテゴリーの過剰付与** — 「舞台がカフェだからカフェカテゴリー」「ホテルの話だからホテルカテゴリー」という理由だけでカテゴリーをつけていないか。施設の体験・空間・文化が主題でない限り業種カテゴリーをそのまま使わない
4. **より適切なカテゴリーの存在** — 選ばれたカテゴリーより正確なものが他にないか（例：カフェ全般より「神社・寺院」のほうが正確な記事もある）

**下書きと一緒に渡すメッセージの形式：**

問題がない場合：
> 📌 **カテゴリー：〔カテゴリー名〕**
> 記事の主題と一致しています。このまま保存します。

変更を提案する場合：
> 📌 **カテゴリーの変更を提案します。**
> 当初の選択：〔元カテゴリー〕
> 提案：〔新カテゴリー〕
> 理由：〔具体的な理由。例：「この施設はカフェが舞台だが記事の主題は禅体験なので、カフェより神社・寺院のほうが正確です」〕
> このまま変更してよいですか？

- カテゴリーが確定してから `frontmatter` に記入してファイルを保存する
- ユーザーが最初に選んだカテゴリーを尊重しつつ、完成した記事の内容と照合して判断する

## 自動生成されるファイル

**gitignore対象（コミット不要）：**
- `public/articles/` — ビルド時に生成
- `public/articles-sitemap.xml` — 記事サイトマップ（記事追加時に自動更新）

**コミット必須（検索機能のため）：**
- `public/pagefind/` — 検索インデックス。**`npm run build` 後は必ずコミットに含めること**
  - gitignore対象から外してあるため、`git add public/pagefind/` でステージングされる
  - コミットしないと Cloudflare Pages に検索インデックスが届かず、検索窓が表示されない

**`public/pagefind/pagefind-entry.json` の非決定的な差分について（2026-07-20確認）：**

- これはPagefindのビルド処理がキー順を保証しないために起きる**非本質的な差分**。`hash`値・`page_count`が変わっていなければ実害はない
- コミット後の検証ビルドでこの差分だけが出た場合は、`git checkout -- public/pagefind/pagefind-entry.json`で元に戻してよい（実質的な変更ではないため再コミット不要）

# ピラー記事生成プロンプト（AEO対応・定義ハブ型）

## 使い方

入力変数（分野・地域・地域の階層・想定読者）を差し替えて使う。
記事執筆開始時の確認事項（git pull・カテゴリ選択等）は通常フローに従うこと。

## プロンプト本文

```
# 役割
あなたは Third Place Japan（TPJ）の編集者です。TPJは家でも職場でもない
「第三の居場所＝サードプレイス」を、独自の7軸基準で評価・認証する日本初の
プラットフォームです。社会学者レイ・オルデンバーグのサードプレイス理論を
土台に、現代日本向けに再解釈した視点で記事を書きます。

# タスク
以下の【分野 × 地域】について、AEO（AI回答エンジン最適化）を前提とした
「定義ハブ ＋ 推薦」両立型のピラー記事を、本文として完成形で生成してください。

# 入力変数（ここだけ差し替える）
- 分野: 〔例：カフェ・スペシャルティコーヒー〕
- 地域: 〔例：表参道〕
- 地域の階層: 〔例：東京都 → 渋谷区/港区 → 表参道エリア〕
- 想定読者: 〔例：ひとり時間を求める20〜40代／出張者／インバウンド〕

# TPJ 7軸評価基準（記事中で必ずこの語彙を使う）
1. 居心地・空間品質（五感品質・素材）
2. 静寂性・プライバシー（静けさ）
3. 特別感・非日常性（ギフト体験）
4. ストーリー・背景への共感（物語の継承）
5. 再訪・継続価値（また来たくなるか）
6. 記録・シェア体験（記録性）
7. インバウンド・多言語対応（外国人適性）

# 絶対ルール
- 実在の店名・施設名は一切出さない。「〇〇通りの角にある築古ビルを改装した一軒」
  のように、立地・空間・素材・客層の「条件と特徴」で具体的に描写する。
- 事実として断定するのは、地域の文化・歴史・地理・街の性格など検証可能な背景のみ。
  個別店の営業情報・価格・評判は書かない。
- 誇大表現を避け、TPJの落ち着いた知的なトーンを保つ。
- AIが引用しやすいよう、各見出し直下の最初の1〜2文で結論を述べる
  （結論先出し）。

# 記事の構成（この見出し順で本文を書く）

## H1（タイトル）
「〔地域〕で〔分野〕をサードプレイスにする：〔地域〕の第三の居場所完全ガイド」
の型を基本に、自然な日本語で。

## リード文（120〜180字）
この地域でこの分野が、なぜサードプレイスとして機能するのかを一段落で要約。
AIがそのまま引用できる密度で書く。

## 1. 〔地域〕という街と「第三の居場所」
- 地域の文化的背景・歴史・街の性格（検証可能な事実のみ）
- なぜこの街が、家でも職場でもない居場所を必要とする／生み出すのか
- 地理的特徴（人の流れ、表通りと裏通り、再開発の有無 等）

## 2. 〔地域〕で〔分野〕が果たすサードプレイスの役割
- この分野がこの街でどんな居場所機能を担うか
- オルデンバーグの中立性・常連・アクセス性の観点で解説

## 3. 〔地域〕の〔分野〕をTPJ7軸で読む
- 7軸それぞれについて、この地域・分野なら何が問われるかを小見出しで解説
- 「こういう空間は居心地軸が高い」と条件ベースで語る（店名なし）

## 4. 〔地域〕で"いいサードプレイス"を見分ける条件
- 読者が自分で良い場所を選ぶためのチェックポイントを、特徴・立地・空間で
- 表通り型／裏通り型、滞在時間別、ひとり/同伴別など切り口を分ける

## 5. 〔地域〕のサードプレイス文化を支える街の文脈
- 関連する地域イベント・季節性・周辺施設の性格（事実ベース）
- この街でしか得られない非日常性・特別感（軸3・4と接続）

## 6. インバウンド視点：〔地域〕の〔分野〕
- 外国人来訪者にとっての分かりやすさ・体験価値（軸7）
- 言語・予約・文化的ハードルの観点

## よくある質問（FAQ・5問）
読者が実際にAIへ尋ねそうな質問を5つ。各回答は2〜3文で結論先出し。
例:「〔地域〕で静かに過ごせる〔分野〕は？」「〔地域〕の〔分野〕の特徴は？」

## まとめ
この街×この分野のサードプレイスの核心を3文で。TPJ認証への自然な言及で締める
（「TPJではこうした空間を7軸で評価・認証しています」程度、宣伝過多にしない）。

# 文字数
本文 2,800〜4,000字。FAQ含む。

# トーン
知的・落ち着いた・断定しすぎない。読者を急かさない。
オルデンバーグ理論への敬意を土台にしつつ、現代日本の生活実感に接続する。
```

## 使用上の注意

- **店名・施設名は一切出さない**のがこのプロンプトの根幹ルール。条件・空間描写で代替する
- 生成後は通常の記事フロー（カテゴリ確認・画像・内部リンク・FAQ書式）を適用する
- `〔地域〕 × 〔分野〕` の組み合わせ数だけピラー記事を量産できる設計
- 地域階層（都道府県→市区町村→エリア）を指定することで、SEOの地名キーワードが自然に埋め込まれる

# 記事品質チェックルール（制作憲法違反パターン）

- ✅ 「有料・要事前確認」

**理由**：料金は変動する情報。誤情報になるリスクがある。

- ✅ 「混雑しやすく、時間帯によっては静寂性が下がる傾向がある」

**理由**：時間帯・季節・イベントによって変動する情報のため。

# JSON-LD実装ルール（Schema.org バリデーション対応・2026-07-02追加）

Schema.org バリデーターで検出された実際のエラー・警告から抽出したルール。
実装・テンプレート編集時に必ず確認すること。

## ✅ Nunjucksテンプレートでの動的テキスト出力（必須パターン）

**禁止**：`"name": "{{ post.data.title }}"` または `| replace('"', '\\"')` で直接出力
- NunjucksがHTML自動エスケープを行い `"` → `&quot;`、`'` → `&#39;` になる
- `<script type="application/ld+json">` 内でも同様に発生する

**正しい実装**：
```njk
"name": {{ post.data.title | dump | safe }}
```
- `dump` = JSON.stringify（適切なエスケープ）
- `safe` = Nunjucksによるさらなるエスケープを防ぐ
- ダブルクォート込みの文字列として出力されるため、`"name":` の後に引用符を書かない

**実際に発生したインシデント（2026-07-20）**：`l3-area.njk`の`hasPart`（関連記事・施設一覧）ブロックで`{{ p["@type"] | dump }}`のように`| safe`が抜けており、L3エリアページ27件のJSON-LDが`&quot;`エスケープで構文崩壊していた（`npm run build`ではエラーが出ないため長期間気づかれなかった）。**`| dump`を書いたら必ずその場で`| safe`が付いているか確認する。** 新しいJSON-LDブロック・一覧系ループを書く／レビューする際は、`grep -n "| dump }}" src/`で`| safe`が欠けた箇所がないか機械的に確認できる（`| dump | safe }}`にマッチしない行が対象）。

---

## ❌ Organization に使えないプロパティ（警告の原因）

以下のプロパティは `Organization` スキーマには不適切なため、使用しない。

| プロパティ | 問題 | 代替 |
|---|---|---|
| `sameAs: []` | 空配列は無意味・警告の原因 | URLが確定するまでプロパティ自体を省略 |
| `knowsAbout: [文字列配列]` | Schema.org は `Thing` オブジェクト型を期待する | 省略するか `Thing` 型で記述 |
| `inLanguage` | `CreativeWork` 用プロパティ。`Organization` には不適切 | 省略する |

**現在の `base.njk` Organization 実装（確定版）：**
```json
{
  "@type": "Organization",
  "name": "Third Place Japan",
  "alternateName": ["TPJ", "サードプレイスジャパン"],
  "url": "...",
  "logo": { "@type": "ImageObject", "url": ".../tpj-logo.png", "width": 900, "height": 275 },
  "description": "...",
  "foundingDate": "2025",
  "areaServed": { "@type": "Country", "name": "Japan" }
}
```
これ以上プロパティを追加する場合は、Schema.org の `Organization` 定義で有効かを事前確認すること。

## ❌ ItemList.ListItem に url プロパティを直接使う / @type:Article を指定する

**禁止①**：`url` を `ListItem` の直接プロパティにする
```json
{ "@type": "ListItem", "position": 1, "url": "https://..." }
```

**禁止②**：`item` に `@type: Article` を指定する
```json
{
  "item": { "@type": "Article", "@id": "https://...", "name": "タイトル", "url": "https://..." }
}
```
→ Google が「Article なら image・author・headline も必要」と警告を出す（2026-07-06 検出）

**正しい実装**：`@type` を省略し `@id`・`name`・`url` のみ記述する
```json
{
  "@type": "ListItem",
  "position": 1,
  "item": {
    "@id": "https://...",
    "name": "記事タイトル",
    "url": "https://..."
  }
}
```
- `@type` を省略しても `ItemList` として有効。余計な型宣言が警告を生む
- 対象ファイル：`src/stories/index.njk`（ItemList ブロック）

## ✅ Article系スキーマの必須フィールド（article.njk / article-certified.njk / article-area.njk）

Article/BlogPosting スキーマでは以下を必ず守ること（2026-07-06 修正・確定）。

**`image` は `ImageObject` で記述する（文字列URL禁止）**
```json
"image": {
  "@type": "ImageObject",
  "url": "https://thirdplacejapan.com/assets/images/articles/xxx.webp",
  "width": 800,
  "height": 450
}
```
- ❌ `"image": "https://..."` — 文字列URLは Rich Results Test で「重大ではない問題」として検出される

**`width`/`height` は実測値を使う。固定値`800/450`のハードコードは禁止（2026-07-20改訂）**
- Unsplashダウンロード時に`w=800`を指定してもアスペクト比は画像ごとに異なり、実際のサムネイルは16:9とは限らない（4:3・3:2等が混在）。`width: 800, height: 450`を全記事に固定していたところ、実測とズレた不正確な値をGoogleに申告し続けていたことが2026-07-20の監査で判明した
- 正しい実装：`.eleventy.js`の`imageDimensions`フィルター（WebPファイルのヘッダーを直接読んでwidth/heightを返す。VP8/VP8L/VP8Xの3形式に対応）を使う
  ```njk
  {% set imgDim = thumbnail | imageDimensions %}
  "image": {
    "@type": "ImageObject",
    "url": "...",
    "width": {{ imgDim.width }},
    "height": {{ imgDim.height }}
  }
  ```
- 適用済みテンプレート：`article.njk`・`article-certified.njk`（`thumbnail or v.image`の解決後の値を渡す）・`article-area.njk`・`about/index.njk`
- **新しく画像を伴うJSON-LD `ImageObject`をテンプレートに追加する場合は、必ず`imageDimensions`フィルター経由にする。`width: 800, height: 450`等の固定値を書かない**

**`publisher.logo` に `width`・`height` を付ける**
```json
"publisher": {
  "@type": "Organization",
  "name": "Third Place Japan",
  "logo": {
    "@type": "ImageObject",
    "url": "https://thirdplacejapan.com/assets/images/main/tpj-logo.png",
    "width": 900,
    "height": 275
  }
}
```

**`author.knowsAbout` は使わない**
- ❌ `"knowsAbout": ["サードプレイス", ...]` — Organization に不適切（文字列配列）
- `author` に含めてよいのは `@type`・`name`・`url`・`description` のみ

## ✅ ItemList の言語フィルタリング（日英混在環境）

**問題**：`collections.articles` には日本語記事・英語記事の両方が含まれる
- 日本語ページ（`/stories/`）の ItemList に英語記事URLが混入してしまう

**実装**：カスタムフィルター `rejectLang` を使用
```njk
{% set recentPosts = collections.articles | rejectLang("en") | head(8) %}
```

**フィルター定義**（`.eleventy.js`）：
```js
eleventyConfig.addFilter("rejectLang", (array, lang) =>
  (array || []).filter(item => item.data && item.data.lang !== lang)
);
```

**注意**：Nunjucks の `selectattr` フィルターはドット記法（`data.lang`）のパス解決に不具合があり使用不可。必ず `rejectLang` カスタムフィルターを使うこと。

### 英語ページ側は `selectLang` を使う（2026-07-13追加）

英語ページ（`/en/stories/` 配下）で「英語記事のみ」を取得したい場合は、`rejectLang("ja")` ではなく専用の `selectLang("en")` を使う。

```njk
{% set enArticles = collections.articles | selectLang("en") %}
```

**フィルター定義**（`.eleventy.js`）：
```js
eleventyConfig.addFilter("selectLang", (array, lang) =>
  (array || []).filter(item => item.data && item.data.lang === lang)
);
```

**背景（2026-07-13発見・修正済みの不具合）**：`src/en/stories/index.njk`（英語記事トップ）が3本の記事カードをハードコードしており、新規に英語記事を追加しても一覧に反映されない状態だった。また `src/_layouts/article.njk` のサイドバー「最新記事」ブロックが `collections.articles | head(5)` と言語フィルタなしで実装されており、英語記事ページなのに日本語記事タイトルが混在して表示されていた。

**恒久対応（今後この不具合を再発させないためのルール）**：
- `src/en/stories/index.njk` は `collections.articles | selectLang("en")` を全件ループで表示する方式に変更済み。**英語記事を追加する際、このページのテンプレートを個別に更新する必要はない**（`lang: en` を設定するだけで自動反映される）
- `src/_layouts/article.njk` のサイドバー「最新記事」は `collections.articles | selectLang(lang or "ja")` でページ自身の言語に絞り込み済み。ラベル・カテゴリ名も `lang == "en"` の場合は英語表記（`postCat.name_en`）に切り替わる
- **新しく「記事一覧」「最新記事」「関連記事」等の一覧系UIを追加する場合は、必ず対象ページの言語で `selectLang` / `rejectLang` によるフィルタリングを行うこと。** `collections.articles` を言語フィルタなしでそのまま使うと同じ不具合が再発する

**🚨 `relatedPostsByCat`・`prevPost`・`nextPost` フィルターも対象（2026-07-19追加・重要）🚨**

上記の「一覧系UI」ルールは `head()` を使う単純な一覧だけでなく、**`relatedPostsByCat`（関連記事）・`prevPost`／`nextPost`（前の記事・次の記事）フィルターにも同様に適用される。** これらは一見「一覧」に見えないため見落としやすく、実際に2026-07-19の監査で `src/_layouts/article.njk`・`article-certified.njk`・`article-area.njk` の計5箇所が未対応のまま放置されており、日本語記事の「関連記事」「前の記事・次の記事」に英語記事が混入する不具合が本番で発生していた（例：`/stories/culture-sports-leisure/koto/` の前後記事・関連記事が英語タイトルで表示される）。

**正しい実装パターン**：
```njk
{# 関連記事 #}
{% set relatedArticles = collections.articles | selectLang(lang or "ja") | relatedPostsByCat(page.url, category_slug, 3) %}

{# 前の記事・次の記事 #}
{% set langArticles = collections.articles | selectLang(lang or "ja") %}
{% set prevArticle = langArticles | prevPost(page.url) %}
{% set nextArticle = langArticles | nextPost(page.url) %}
```

**理由**：`collections.articles` は日英混在の全記事を日付降順でソートしたコレクション（`.eleventy.js` の `addCollection("articles")`）。`prevPost`/`nextPost` はこの配列上の位置（インデックス）で前後を決めるため、日本語記事の直前直後にたまたま英語記事が公開されていると、そのまま前後記事リンクとして表示されてしまう。`relatedPostsByCat` も `category_slug` 一致のみで絞り込み、言語は見ないため、同カテゴリに英語記事があれば関連記事に混入する。

**チェック方法**：新しいレイアウト・テンプレートを追加/編集する際は、`collections.articles` を直接（`selectLang`/`rejectLang` を挟まずに）参照している箇所がないか `grep -n "collections\.articles" src/_layouts/ src/stories/ src/en/` で確認する。`sitemap.njk`（全言語を含めるべきサイトマップ）のような例外を除き、原則すべて言語フィルタを挟むこと。

## ✅ LocalBusiness @type のカテゴリ別マッピング（venue.njk・2026-07-09追加）

`hasMenu`・`servesCuisine` プロパティは Schema.org 上 `FoodEstablishment` 系の型にしか存在しない。汎用の `@type: "LocalBusiness"` のままこれらを出力すると、Schema.orgバリデーターが「プロパティが認識されない」と警告する（GBCの`hasMenu`で実際に検出・修正済み）。

**実装（`venue.njk` 冒頭で定義）：**
```njk
{%- set foodEstablishmentTypes = {"cafe-specialty-coffee": "CafeOrCoffeeShop", "bar-whisky-cocktail": "BarOrPub"} -%}
{%- set businessType = foodEstablishmentTypes[venue.category_slug] or "LocalBusiness" -%}
```
`LocalBusiness` JSON-LD本体（`@type`）と、Review内の `itemReviewed.@type` の**両方**に `{{ businessType | dump | safe }}` を使う（2箇所とも揃える）。

**ルール：**
- `venue.category_slug` が食・飲料系（カフェ／バー等）の場合のみ、Schema.orgの `LocalBusiness > FoodEstablishment` のサブタイプに切り替える
- 該当しない業種（ホテル・スパ・文化施設等）は従来通り `LocalBusiness` のまま
- 新しい食・飲料系カテゴリを追加する場合（レストラン等）は `foodEstablishmentTypes` にマッピングを追加する（例：`"restaurant-dining": "Restaurant"`）。追加前に対象の型がSchema.orgで現行有効か（廃止されていないか）を確認すること
- `hasMenu`・`servesCuisine` 以外の食関連プロパティを追加する場合も、必ず対象の `@type` がそのプロパティをサポートするか事前に確認する

## ✅ LocalBusiness の検索最適化プロパティ（venue.njk グレード条件付き）

以下のプロパティはグレードに応じて条件出力する（`thisLevel` による制御）。

### openingHours（Silver以上・`thisLevel >= 2`）

```json
"openingHours": ["Mo-Su 10:30-20:00"]
```

- Google・AIが「今開いているか」を判定する。形式は Schema.org の `OpeningHoursSpecification` 準拠
- 曜日コード：Mo / Tu / We / Th / Fr / Sa / Su
- 複数パターン例：`["Mo-Fr 09:00-18:00", "Sa-Su 10:00-17:00"]`
- `venues.json` には `opening_hours`（配列・JSON-LD用）と `opening_hours_display`（文字列・UI表示用）の2フィールドで管理する

### sameAs（Silver以上・`thisLevel >= 2`）

```json
"sameAs": ["https://example.com/", "https://www.instagram.com/handle"]
```

- 公式サイト・Instagram・Googleビジネスプロフィール等のURLを配列で列挙する
- エンティティ認識を強化し、AIが同一施設として正しく識別する効果がある
- 空配列 `[]` は警告の原因になるため、URLが確定しない場合はフィールド自体を省略する

### amenityFeature（Platinum以上・`thisLevel >= 4`）

**禁止**：文字列配列で出力する
```json
"amenityFeature": ["Wi-Fi", "電源", "禁煙"]
```

**正しい実装**：`LocationFeatureSpecification` オブジェクト形式
```json
"amenityFeature": [
  { "@type": "LocationFeatureSpecification", "name": "Wi-Fi", "value": true },
  { "@type": "LocationFeatureSpecification", "name": "電源", "value": true },
  { "@type": "LocationFeatureSpecification", "name": "禁煙", "value": true }
]
```

- `venues.json` の `amenity_features` は文字列配列で保持し、テンプレート側でオブジェクト変換する
- `value: true` は「その設備がある」を意味する。ない場合は `value: false` または省略する
- 「Wi-Fiあるカフェ」「電源あり」等の条件検索・AI回答に対応する重要プロパティ

### smokingAllowed（Platinum以上・`thisLevel >= 4`）

```json
"smokingAllowed": false
```

- `true` / `false` のBoolean値で指定。文字列 `"false"` は不可
- 禁煙・喫煙可の条件検索に対応する

### geo（グレード非依存・`venue.lat and venue.lng` があれば全グレード出力）

**`geo`（GeoCoordinates）はこのセクションの他プロパティ（openingHours・sameAs・amenityFeature等）と異なり、グレード（`thisLevel`）に関わらず`venue.lat`・`venue.lng`が設定されていれば全施設で出力する。** TPJセレクト（`grade_slug: "select"`、`thisLevel = 0`）も対象に含まれる。

```njk
{% if venue.lat and venue.lng %},
"geo": {
  "@type": "GeoCoordinates",
  "latitude": {{ venue.lat }},
  "longitude": {{ venue.lng }}
}
{% endif %}
```

**理由（2026-07-21修正）**：位置情報は「掲載グレードが上がるほど開示される拡張プロパティ」ではなく、施設を一意に識別するための基本情報。以前は他のグレード限定プロパティと同じ`{% if thisLevel >= 1 %}`条件を流用していたため、TPJセレクト19件（当時）のJSON-LDから`geo`が一律で欠落していた。地図・位置情報を伴うAI検索・AEOにおいて`geo`は重要度が高く、グレードで出し分ける対象から除外する。

### geo（lat / lng）座標の取得ルール（必須・絶対厳守）

`venues.json` の `lat` / `lng` は **店舗の住所そのものの座標** を設定すること。

**取得方法（正）**：
1. Googleマップで店舗名を検索して店舗ページを開く
2. URLに表示される `@緯度,経度` の値をそのまま使用する
   - 例：`@35.6775167,139.7078346,17z` → `"lat": 35.6775167, "lng": 139.7078346`
3. または店舗ページの「共有」→「地図を埋め込む」URLから座標を取得する

**禁止**：
- ❌ エリアや最寄駅の概算座標を入れる（例：千駄ヶ谷エリアの中心点など）
- ❌ 住所から手動で推定した座標を入れる
- ❌ 小数点を丸めた概算値（例：`35.659`）を使う

**理由**：`lat`/`lng` は JSON-LD `geo: GeoCoordinates` として出力される。

### maps_url フィールド（必須・絶対厳守）

**❌ `?q=lat,lng` 形式を「地図を見る」ボタンに使ってはいけない**

```
https://maps.google.com/?q=35.6775167,139.7078346
```
→ Googleが最寄りの登録店舗に座標をスナップするため、**意図しない別店舗にピンが立つ**場合がある（2026-07-06 GBCで確認：TERRAFORM COFFEEにピンが立った）

**✅ 正しい実装：`maps_url` フィールドに店舗名＋住所のクエリURL を設定する**

```json
"maps_url": "https://maps.google.com/?q=店舗名+住所"
```

例：
```json
"maps_url": "https://maps.google.com/?q=グリーンビーンズコーヒー+東京都渋谷区千駄ヶ谷3-30-1"
```

**禁止**：`/maps/place/店舗名/@緯度,経度` 形式（URLパス部分の日本語はブラウザで脱落する場合がある。2026-07-06確認）

**取得・設定方法**：
1. `?q=店舗名+住所` の形式で組み立てる（店舗名のみでも可。住所を加えると精度が上がる）
2. ブラウザで開いて正しい店舗パネルが表示されることを目視確認する
3. `venues.json` の `maps_url` フィールドに設定する

**「地図を見る」ボタンのURL優先順位（venue.njk）**：
1. `venue.maps_url` が設定されている → そのURLを使用（推奨）
2. 未設定 → `https://maps.google.com/?q={{ venue.address }}` （住所検索にフォールバック）

**確認方法**：設定後にブラウザで実際に開き、目視で正しい店舗にピンが立つことを確認する。

# SEO・AEO記事作成指針

## メディア名の表記ルール（SEO対策・必須）

| 表記 | 使い方 |
|------|--------|
| **Third Place Japan** | メインの表記。記事冒頭・見出し・まとめ・Q&Aなど重要な箇所で使う |
| **サードプレイスジャパン** | 日本語検索ヒット用。本文中に自然な形で数回使う（1記事あたり3〜5回を目安） |
| **TPJ** | 同一段落内での2回目以降の言及など、繰り返しを避けたい箇所に使う |

## サイト内の呼称ルール（検索されない造語を避ける・2026-07-20追加）

**禁止**：「TPJガイド」「TPJ Guide」のように、TPJの略称と一般語を組み合わせただけの独自呼称を、ナビ・フッター・見出し・パンくずなど**サイト全体で繰り返し使う固有の呼び名**として採用すること。

**理由**：日本人ユーザーは「サードプレイス」（カタカナ）で検索し、外国人ユーザーは「third place」（英語）で検索する。「TPJガイド」はどちらの検索行動にも一致しない、認知度ゼロの造語であり、SEO・AEOの両方で機会損失になる。

**ルール**：サイト内の呼称・カテゴリ名・セクション見出しには、検索される語（日本語なら「サードプレイス」、英語なら「Third Place」）を必ず含める。TPJという略称自体を使うこと自体は問題ないが、それ単独・それを核とした造語を検索導線になる呼称の主役にしない。

**適用例（2026-07-20修正）**：`/stories/about/`（サードプレイスの定義・8条件・TPJ7軸評価基準などを扱うピラー記事群）の呼称を「TPJガイド」から「サードプレイスガイド」に統一。対象：グローバルナビ・フッター・パンくず（画面表示＋JSON-LD）・ページtitle/H1・メインサイトのFAQリンク。英語文脈で同種のラベルを使う場合は「TPJ Guide」ではなく「Third Place Guide」とする。

**注意**：既存記事本文中で個別記事のタイトルの一部として「TPJ Guide」を含むもの（例：英語記事タイトル "What Is a Third Place? The TPJ Guide to..."）は対象外。タイトル冒頭に "Third Place" が既にあり検索語を満たしているため、無理に置き換えない。このルールが対象とするのは、検索語を一切含まない呼称がサイト全体の導線・見出しとして使われているケースである。

## タイトル作成のルール

記事タイトルはSEO対策の中で最も重要な要素。以下の3つのキーワードグループを**必ず自然な形でタイトルに含める**こと。

**① 業種ワード（記事のカテゴリに対応する業種を1つ）：**
- カフェ・コーヒー / ホテル・ラウンジ / 温泉旅館 / レストラン / バー / スパ・サウナ / コワーキング / 神社・寺院 / 自然・グランピング 等
- 例：「東京のスパで整う」「京都の神社でリトリート体験」

**② 地名（1〜2つ）：**
- 東京 / 日本 / 各都市名（京都・大阪・北海道等）/ 地域名
- タイトルに地名が入ることで「エリア検索」にヒットしやすくなる

**③ 体験・目的ワード（「サードプレイス」は原則必須・2026-07-20改訂）：**
- **「サードプレイス」は日本人ユーザーが実際に検索する語であり、原則としてタイトルに含める**（上記「サイト内の呼称ルール」と同じ理由）。「地域名＋業種＋ガイド」型のタイトルでは「〇〇のサードプレイスガイド：△△編」のように業種ワードと組み合わせて自然に挿入する
- 居場所 / 第三の場所 / 認証 / 体験 / インバウンド は無理に入れず、記事の内容と自然に結びつく場合のみ使用する
- 施設名の断定・具体名を避ける等の他ルールと矛盾する場合や、TPJの制度名（例：認証グレード解説記事）が主題の場合も、可能な限り「サードプレイス」を導入部の一語として添える

**タイトル作成のルール：**
- 「〜してみた」「〜に行ってきました」「〜が最高だった」など体験型タイトルはAIに引用されやすい
- 30〜35文字以内を目安（長すぎると検索結果で切れる）。ただし「地域名のサードプレイスガイド：業種編。キャッチコピー」のような二段構成タイトルは、既存の運用上この目安を超えることが許容されている（40〜52字程度）
- 検索意図を想定して立てる（「東京 サードプレイス おすすめ」「京都 カフェ 居場所」「スパ 整う 東京」など）
- 下書きを書き始める前に、SEO/AEOを意識したタイトル案を2〜3個提示してユーザーに選んでもらう
- **新規記事の執筆時だけでなく、既存記事のタイトルを見直す際も「サードプレイス」または「第三の居場所」がタイトルに入っているか確認する。** 2026-07-20の監査で、地域×業種ガイド型12記事のタイトルにこの語が一つも入っていないことが判明し、是正した（詳細は実装ログ参照）

## 施設ページ（venue）のtitleタグルール（2026-07-09追加・2026-07-09改訂）

施設詳細ページ（`/stories/select/{slug}/`）の `<title>` は `src/stories/select/venue.njk` の `eleventyComputed.title` で**全施設共通のテンプレート**として一括生成する。個別施設ごとにfrontmatterで上書きしない。

**確定フォーマット：**
```
{{ venue.name }}（{{ 都道府県（都/道/府/県を除去） }}・{{ 市区町村（区/市/町/村を除去） }}）| {{ venue.category }}
```
実装（`src/stories/select/venue.njk`）：
```njk
title: "{{ venue.name }}（{{ venue.area_primary.prefecture | replace('都','') | replace('道','') | replace('府','') | replace('県','') }}・{{ venue.area_primary.city | replace('区','') | replace('市','') | replace('町','') | replace('村','') }}）| {{ venue.category }}"
title_suffix: "Third Place Japan"
```
出力例：
```
Green Beans Coffee（東京・渋谷）| カフェ・スペシャルティコーヒー | Third Place Japan
```

**ルール：**
- エリア名は**都道府県＋市区町村（例：東京・渋谷）**を使う。**個別のエリア名（千駄ヶ谷等）は入れない**（2026-07-09改訂）。理由：エリア名は検索ボリュームが小さく、titleを伸ばすだけでSERP表示が見切れやすくなる一方、都道府県＋市区町村（特に「渋谷」等の主要区名）は検索ボリュームが大きく実効性が高い。titleタグはGoogleに全文インデックスされるため精度の高い上位語を優先する
- 業種（`venue.category`）を必ず含める（検索流入・地名キーワード対策）
- **「No.1」「最高」「一番」など裏付けのない優劣・ランキング表現は禁止**（上記「記事品質チェックルール」の誇大表現禁止と同じ理由。TPJに施設間の公式ランキング制度は存在しない）
- H1（`venue.name`）や`citation`など他の表示箇所とは独立した項目であり、titleタグ変更がページ本文・パンくず表示に影響することはない
- テンプレートが共通のため、変更すると**全施設ページのtitleが一括で変わる**。個別施設だけ変えたい場合はテンプレートではなくvenues.jsonにtitle専用フィールドを追加する設計に変更する必要がある（現状は未対応）
- `title_suffix: "Third Place Japan"` を設定し、末尾を（他ページ共通の）「Third Place Japan Stories」より短い「Third Place Japan」にしている（下記「titleタグの末尾ブランド表記（title_suffix）」参照）。titleが長くなりがちな施設ページ特有の対応

## titleタグの末尾ブランド表記（title_suffix）とブランド名重複禁止ルール（2026-07-09追加）

`src/_layouts/base.njk` の `<title>`・`og:title`・`twitter:title` は共通して以下の形式で生成される：
```njk
{% if title %}{{ title }} | {% endif %}{{ title_suffix or "Third Place Japan Stories" }}
```
- `title_suffix` を明示的にfrontmatter/eleventyComputedで設定しない限り、末尾は自動的に「Third Place Japan Stories」になる
- ページごとに末尾のブランド表記を短くしたい場合（施設ページ等）は、個別に `title_suffix` を設定する（例：`title_suffix: "Third Place Japan"`）。値を変更しても他ページには一切影響しない

**🚨 禁止事項（ブランド名重複バグの再発防止・絶対厳守）🚨**

**`title:` frontmatterの値自体に「Third Place Japan」「Third Place Japan Stories」などのブランド名を含めてはいけない。** `base.njk` が末尾に自動付与するため、frontmatter側にも書くと「〇〇 | Third Place Japan Stories | Third Place Japan Stories」のように**ブランド名が二重表示される**。

- ❌ `title: "Third Place Japan Stories | 場所について、読む。"`
- ✅ `title: "場所について、読む。"`（自動的に「場所について、読む。| Third Place Japan Stories」になる）

2026-07-09に以下4ページで実際にこのバグが発生し修正済み：`src/stories/index.njk`／`src/stories/latest.njk`／`src/stories/about/certification-grades/index.njk`／`src/en/stories/index.njk`。新規ページ作成時・title編集時は必ずこのルールを確認すること。

## 必須キーワード（全記事に含める）

- **「サードプレイス」または「第三の居場所」** → 全記事に必ず1回以上
- **「日本」** → 全記事に必ず1回以上。「日本のサードプレイス」「日本初の認証」「日本全国」など文脈に合う表現で使う
- **「東京」または対象施設の都市名** → 全記事に必ず1回以上。東京の施設なら「東京カフェ」「東京のサードプレイス」、京都なら「京都の旅館」など対象地に合わせる
- **業種キーワード** → 記事の対象に応じてカフェ・ホテル・スパ等を自然に含める

## 地名キーワードの挿入戦略（SEO・AEO最重要）

検索ボリュームが大きい地名キーワードは全記事に必ず自然な形で入れること。

- **施設の所在地名** → 全記事に必ず1回以上。「渋谷区」「新宿」「京都・嵐山」など正確な地名を使う
- **最寄り駅・アクセス情報** → まとめ・FAQ・アクセス説明など文脈に合う場所に挿入。「〇〇駅から徒歩5分」「〇〇空港から1時間」など
- **広域地名** → 「東京都内」「関東」「近畿」「九州」など広域ワードを文脈に合わせて追加（全記事8割を目安）
- 不自然な詰め込みは禁止。あくまで文章の流れの中に溶け込ませる

## 周辺スポット・観光キーワード（SEO・AEO対策）

施設周辺の観光スポット・交通ハブ・自然・文化施設などのワードが少ない記事には自然に追加すること。

- 全記事の7割を目安に含める（無理に詰め込まない）
- FAQ回答・まとめ・アクセス説明など、文脈に合う場所に1〜2ワード程度
- 例：「〇〇神社の近く」「△△公園の帰り道に」「□□美術館からアクセスしやすい」
- 既存の文章に溶け込む形で追加し、不自然にならないよう注意する

## FAQ（Q&A）の書き方

- AIや音声検索が「そのまま読み上げられる」簡潔な回答を心がける
- 質問は実際に検索されうる形で立てる。例：
  - 「東京でおすすめのサードプレイスは？」
  - 「TPJ認証とはどういう意味ですか？」
  - 「第三の居場所として使えるホテルはありますか？」
- 回答は2〜3文に収める。専門性と信頼性を感じさせる語尾（「〜です」「〜できます」）で締める
- 回答の冒頭でそのまま答えを述べる（前置き不要）。例：「Third Place Japanが認証した〇〇がおすすめです。〇〇駅から徒歩5分、7軸基準で高評価を獲得した施設です。」

## 内部リンク（AEO対策・毎回必須）

新記事を作成するたびに、以下の2つを必ず行うこと：

1. **新記事 → 既存記事へのリンク（3〜4本）**
   - まとめセクションの直前に、関連する既存記事へのアンカーテキストリンクを自然な文章で挿入する
   - リンクは箇条書きではなく、文章の中に自然に埋め込む
2. **既存記事 → 新記事への逆リンク（2〜3本）**
   - 新記事と関連する既存記事の適切な箇所に新記事へのリンクを追記する
   - 既存記事のまとめ直前など、文脈に合う場所に1文追加する形で挿入する

ユーザーからの指示がなくても、記事作成の仕上げとして自動的に実施すること。

**「3〜4本」はリンク数ではなく異なるリンク先の本数を指す（2026-07-20明記）**：2026-07-20の監査で、同じリンク先を2回貼って合計本数だけ3本にしている記事（実質2記事にしかリンクしていない）が2件見つかった。内部リンクを数える際は`<a href>`の出現回数ではなく、ユニークなURL数でカウントすること。

## 構造変更時のリンク健全性チェック（2026-07-20追加・運用ルール）

**孤立ページ・リンク切れは、`venues.json`／`areas.json`のスラッグ変更・エリア追加・カテゴリ再編のたびに「ビルドエラーにならない形」で静かに発生する。** `npm run build`はテンプレートが正常にレンダリングできれば成功と判定するため、「あるURLへのリンクが実は存在しないページを指している（リンク切れ）」「あるページが現在のデータ・テンプレートの組み合わせではもう生成されず、他のどこからもリンクされていない（孤立ページ）」のどちらも検知しない。2026-07-20に`public/stories/area/tokyo/shibuya-ku/daikanyama/luxury-hotel-lounge/index.html`が孤立ページ化していたことに気づいたのも偶然の目視発見であり、機械的なチェック工程が存在しなかったことが原因だった。

**実施タイミング（以下のいずれかを行った直後に必須）：**
- `src/_data/areas.json` へのエリア追加・削除・`slug`変更
- `src/_data/venues.json` の `slug` 変更・大量追加
- `src/_data/categories.json` のカテゴリ追加・削除・`slug`変更
- 記事の`permalink`変更、または大量の記事に影響するレイアウト（`src/_layouts/`・`src/_includes/`）の構造変更

**チェック内容（`npm run build`実行後に行う）：**
1. **リンク切れチェック**：`src/`配下の全記事・全テンプレートから`](/...)`形式の内部リンクを抽出し、リンク先が`public/`配下に実際に生成されているか（`index.html`が存在するか）を機械的に検証する
2. **孤立ページの簡易検出**：`public/`配下の主要ディレクトリ（`stories/area/`・`stories/select/`等、データ駆動で生成されるページ群）について、そのURLを指す内部リンクが他のどこか1箇所にも存在しないページがないか確認する。完全な自動化は難しいため、「直近の構造変更で影響を受けたはずの範囲」に絞って重点的に確認する

**検証方法の例（2026-07-20フェーズ4で実施した手法）**：Node.jsスクリプトで対象記事の`](/...)`リンクを抽出し、リンク先ファイルが`public/`に実在するかをファイルシステムで確認する。孤立ページ側は、`public/`配下の生成URL一覧と、全記事・全テンプレートから発リンクされているURL一覧を突き合わせ、差分を洗い出す。

**このルールが直接埋めた未対応事項**：2026-07-09の実装ログに残っていた「同様の孤立ページが他にも残っている可能性があり、別途棚卸しが必要」という未対応メモは、2026-07-20のフェーズ4監査（内部リンク再点検）で棚卸しを実施し消化済み。今後は特定の一度きりの棚卸しではなく、構造変更のたびに行う定常運用として位置づける。

# 文体ガイドライン（TPJ編集部）

**大前提：権威ある第三者評価の視点で書く**
TPJ編集部は「広告媒体」ではなく「認証機関」。店舗の宣伝ではなく、独立した評価者として書くこと。

- ✅「〜と評価しました」「〜を認証しました」「〜の観点から検証した結果」
- ✅「編集部が実際に訪問し、7軸基準で評価しました」
- ❌「〜が素晴らしいです」「ぜひ行ってみてください」（宣伝口調）
- ❌「〜のようです」「〜と思います」（権威を弱める曖昧な表現）

**文体の特徴：**
- 簡潔・明確な文章（ミシュランガイドのような簡潔さ）
- 「です・ます」調を基本に、必要に応じて体言止めを使う
- 感情的表現より客観的評価を優先
- 読点（、）は必要最低限に抑える

## お金・ビジネス色を排した表現ルール（2026-07-06追加）

TPJセレクト・認証グレード関連のページ・記事では、金銭や商取引を連想させる表現を避け、「体験・空間・文化」の文脈で語ること。

**禁止表現（お金・商取引をイメージさせる語）：**
- ❌ コンサルティング / コンサルタント
- ❌ 提供しています / ご提供
- ❌ サービスとして / 有償で / 料金
- ❌ ご契約 / お取引 / ご発注

**推奨表現（体験・プロデュース文脈）：**
- ✅ 空間プロデュース / 居場所づくり
- ✅ 取り組んでいます / 実践しています
- ✅ 知見をもとに / 実証してきた
- ✅ ともに考える / 空間を育てる

**適用範囲：**
- `src/_data/venues.json` の `citation` / `faq` / `body` フィールド
- `src/_layouts/venue.njk` の静的コピー（実証店舗セクション・底部テキスト等）
- 記事本文・FAQ（特にFlagship・認証関連の文章）
- **メインサイト `public/index.html` の `#services`（法人向けサービスメニュー）セクション全体（2026-07-09追加）**：テーブル本文・対象事業者欄・独立性注記・各サービスの詳細モーダル本文を含む。「商品化」「収益源」「実務支援」「運用支援」「ご提供」「承ります」等の商取引寄り・お金寄りの語は「取り組みます」「サポートします」等に置き換える

## TPJセレクト・認証グレードページの文字色ルール（2026-07-06追加）

TPJセレクトおよびTPJ認証グレード関連ページ（施設詳細・FAQ・実証店舗セクション等）の本文テキストは原則 **こげ茶（`#3D2B1F`）** を使用する。

| 対象箇所 | カラー | 実装方法 |
|---|---|---|
| 施設情報カード（営業時間・価格帯・電話） | `#3D2B1F` | `style="color:#3D2B1F;"` |
| FAQアコーディオン 質問文（`<summary>`） | `#3D2B1F` | `style="color:#3D2B1F;"` |
| Flagship 実証店舗セクション 本文段落 | `#3D2B1F` | `style="color:#3D2B1F;"` |
| `.prose p`（記事本文段落） | `#3D2B1F` | `base.njk` の `.prose p` に `color: #3D2B1F;` |

**理由：** 黒（`#1C1C1C`）は主張が強く、TPJの「静か・品がある」トーンに合わない。こげ茶は温かみがあり、権威と親しみのバランスを保つ。

**注意：** `text-dark`（`#1C1C1C`）を施設ページの本文テキストに使うことは禁止。見出し（`h1`/`h2`）・施設名など「強調が必要な箇所」のみ `text-dark` を許可する。

**構成のルール：**
- **冒頭に「答えリード」を1〜2行置く（AEO必須）**：記事の主旨・結論を太字で簡潔にまとめる。AIが150字以内に直接回答を抽出できるようにする
  ```markdown
  **〇〇は△△として認証されています。□□の観点で〜。**

  （詳細な評価・体験の本文）
  ```
  **既存記事への遡及適用ルール（2026-07-20追加）**：このルールは新規記事だけでなく全記事に適用される。2026-07-20の全記事監査で、初期に作成された「業種×エリア」記事32本（バー・カフェ・コワーキング・ホテルラウンジ・会員制ラウンジ・高級サロン・レストランの各カテゴリ）に答えリードが一件も無いことが判明し、遡及的に追加した（実装ログ参照）。**新しいカテゴリ・記事バッチを大量生成した直後は、このルールを含む必須ルールが後から追加されたルールも含めて全件に適用されているか、定期的に機械監査すること。** 監査方法：各記事本文冒頭300〜600字以内に`**〜**`（15字以上の太字）が存在するかをスクリプトでチェックする。
- **各H2セクション冒頭にも結論文を1文置く（AEO強化・2026-07-02追加）**：記事冒頭だけでなく、各セクションの見出し直下の1文もAIが引用する対象になる。以下のルールに従う：
  - ✅ 結論先出し型：「〇〇は〜である。」「〜が最も有効な指標だ。」
  - ❌ 禁止パターン①（How-To型）：見出し直下を `**箇条書きキーワード**：` から始める（結論が後置される）
  - ❌ 禁止パターン②（問い形）：「〜はどのように〜か。」で始める
  - ❌ 禁止パターン③（否定形）：「〜ではありません。」で始める
  - ❌ 禁止パターン④（シーン描写型）：感覚的描写・情景描写から始める
  - 特に「選び方・見分け方」系セクションは How-To型に陥りやすい。「〇〇を選ぶには〜という3つの観点が有効だ。」のように結論→根拠の順で書く
- まとめセクションは任意（執筆開始時に確認）だが、AI引用の要約抽出ポイントとして機能するため付けることを推奨する

### description文字数ルールの改訂（115〜160字 → 110〜120字）

**対象ファイル**
- `CLAUDE.md`（フロントマターのテンプレート／ピラー記事生成プロンプトの2箇所）
- `docs/TPJ制作憲法.md`（5章の構成規定／7章セルフチェックリストの2箇所）

**変更内容**：上記監査で`concept/`配下23記事のdescriptionが115字未満と判明したことを受けて基準を再検討した結果、**120字＝検索結果・OGPカード等でdescriptionが途切れずに全文表示される目安（特にPC表示）**という表示上の制約を優先し、110〜160字だった許容幅を110〜120字に縮小した。160字まで許容すると表示時に末尾が「…」で切れ、一文目で結論を完結させても後半の補足が読者に見えないケースが生じるため。

**理由**：情報密度（AI単体引用に耐えるための下限110字）と表示の完全性（切れずに見える上限120字）の両方を満たす範囲に絞り込んだ。上限を厳しくした分、二文目以降を削るなど本文の圧縮が必要になる記事が増える見込み。

**注意**：この改訂は2026-07-19時点で新規作成・改稿する記事から適用する。既存記事（`concept/`配下23記事など）の一括改稿は、下記「`concept/`配下24記事のdescription一括改稿」で同日中に対応済み。

{% if item.city.slug == 'shibuya-ku' and item.prefecture.slug == 'tokyo' %}

## TPJ ピラー記事 制作ルール

TPJ（Third Place Japan）の「地域 × 15分野」ピラー記事を制作する際の要点。
**詳細な制作憲法は `docs/TPJ制作憲法.md` を必ず参照すること**（このセクションは要約）。

### 参照ファイル
- 制作憲法（不変ルール全文）：`docs/TPJ制作憲法.md` ← 記事制作時は必読
- 記事生成の実行手順：`docs/記事生成プロンプト.md`
- 入力データ（1行＝1記事）：`data/TPJ_東京ピラーマス表.xlsx`
- 記事出力先：`articles/{分野slug}-{地域slug}.md`

### 絶対に守る5原則（詳細は制作憲法）
1. **店名・施設名を一切出さない**。立地・空間・素材・客層の「条件」で描く。
2. **断定してよいのは検証可能な地域情報のみ**（地理・歴史・交通・統計）。個別店の営業・価格・評判は断定しない。
3. **AI臭を出さない**。「近年」「いかがでしたか」等の定型句禁止。現地を知る編集者の温度で書く。
4. **7軸を背骨にする**（居心地／静寂性／特別感／ストーリー／再訪／記録性／インバウンド）。
5. **重複させない立体構造**。都(幹)→自治体(面)→密集エリア(点)。子は親を繰り返さず内部リンクで委譲。各記事は「その地域×分野でしか書けない核」を3つ以上持つ。

### 制作フロー
1. 入力＝マス表の該当行（優先度A→B→Cの順に着手）。
2. `docs/記事生成プロンプト.md` の STEP1〜3 に従う（先に「固有の核」3点を言語化）。
3. 構成・文体・リード規定は制作憲法に従う（リードは最大160字・一文目で結論完結）。
   metadescriptionは110〜120字（120字＝PC表示等で途切れずに見える目安。上記「フロントマターのテンプレート」参照）。断定可能な地域情報は言い切り、「〜傾向があります」多用禁止。
4. 出力前に制作憲法 第7章のセルフチェック**13項目**を実行。**全項目クリアまで確定しない**。
   （追加3項目：Markdown構文・metadescription字数・JSON-LD改行なし確認）

### このセクションの優先順位
TPJ記事制作に関しては、このルールと `docs/TPJ制作憲法.md` を最優先で参照する。
矛盾を感じたら制作憲法（docs）の記述に従う。
