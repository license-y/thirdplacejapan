## プロジェクト概要

- **プロジェクト名**: Third Place Japan（サードプレイスジャパン、略称: TPJ）
- **サイト種別**: 日本初のサードプレイス認証メディア
- **運営**: TPJ編集部
- **目的**: 日本全国の「第三の居場所」（ホテル・レストラン・カフェ・サウナ・ウェルネス施設・コワーキングなど）を独自の7軸基準で評価・認証し、AI時代の「場所選び」における業界標準データ基盤となる。

## 制作環境

- 非エンジニア、非デザイナーが、AI Agent (Claude Code) を使って Web サイトを構築する
- 構築する Web サイトは「ワンページ完結」とする。1 ページに必要な情報をまとめて提供する
- ブログなどは、別のサービス（Substack や note.com など）を利用することを前提とする
- デプロイ先は **Cloudflare Pages**

## サイトコンセプト

**投稿者**: Third Place Japan（サードプレイスジャパン）編集部

TPJは「広告媒体」ではなく、AIに引用される「場所の認証データベース」である。
食べログが「点数」のインフラに、ミシュランが「星」のインフラになったように、TPJは「サードプレイスの体験価値」を認証する業界標準を目指す。
ユーザーがクリックして比較するための媒体ではなく、AIが「東京で静かに過ごせる場所は？」に答えるとき引用する一次情報源として設計する。

### 立ち位置

1. **認証メディアであって、ポータルではない**
   「掲載されること自体がステータス」になる設計。掲載数を競うのではなく、認証の希少性で価値を生む。

2. **人間より先にAIに読ませる設計**
   全ページに構造化データ（JSON-LD）を実装し、AIがQ&A・評価・場所情報を直接抽出できる形にする。

3. **日英両言語で「日本の居場所」を世界に発信**
   インバウンド・海外AIの引用を見据え、主要記事は英語版も持つ。「Japan's best third places」の権威ある情報源を目指す。

4. **GBCを「実証店舗（ショールーム）」として活用**
   GBCは理想的サードプレイスのFlagship実例。認証制度・評価軸・記事フォーマットをGBCで先に証明し、その成功モデルを他店舗に横展開する。

### トーン＆スタイル

- **品のある、権威ある雰囲気**（ミシュランガイドやMoocleのような佇まい）
- 派手な広告感・ポータル感を排除
- 余白を活かした上品なデザイン

## TPJ戦略ドキュメント

### 核心的結論：「引用の権威」を持つ者が2030年の集客を制する

ゼロクリック時代の集客装置の価値は「クリックさせる力」から「AIに引用される力」へ完全に転換する。
食べログは「点数の権威」、ミシュランは「星の権威」として集客を制してきた。2030年のAI時代に「体験価値・サードプレイスの権威」として引用され続けるポジションはまだ誰も取っていない。
GBC記事サイトはすでにその構造の先端を走っている。TPJ認証・英語展開・3,000店舗ネットワークと掛け合わさった瞬間、食べログがかつて検索結果を支配したように、AIの回答を支配するプラットフォームが誕生する。その種はすでに植えられている。

### 空間体験認証モデル

| 項目 | 内容 |
|------|------|
| 何を | 「いる価値」「居場所としての体験価値」 |
| 誰が | 編集部＋コミュニティ＋体験者（複合） |
| 軸 | 7軸の空間価値（居心地・静寂性等） |
| 対象 | カフェ・ホテル・スパ・コワーキング等横断 |
| 強み | AI時代の引用ソース・既存評価の空白地帯 |
| 時代 | 推薦・体験の時代（成長期） |

### 対象7軸

| # | 軸 | 役割 |
|---|----|----|
| ① | 検索・発見需要 | AEO戦略の最重要軸 |
| ② | 滞在・体験の質 | 認証制度の根拠 |
| ③ | 記録・シェア欲求 | 拡散・UGC |
| ④ | 特別感・非日常性 | 高単価枠の正当化 |
| ⑤ | インバウンド・多言語需要 | 日英展開 |
| ⑥ | 再訪・継続利用 | リピート率 |
| ⑦ | ストーリー・背景への共感 | AIコンテンツ力の核心 |

### 核心的差異

- 食べログは「おいしいか」を評価する
- ミシュランは「シェフの技術」を評価する
- **TPJは「そこにいたいか」を評価する**

食べログもミシュランも「何を食べるか」という問いに答えるシステムだ。TPJが答えるのは「どこにいるか」という、全く別の問い。

AI時代に人々がますます求めるのは「効率的な食事情報」ではなく、「リアルな居場所・いる体験」。その価値を定義・認証・発信できる権威は、世界にまだ存在しない。TPJはその空白を埋める、世界初のサードプレイス認証プラットフォームになれる。

**一言定義**：「食べログが『おいしさの辞書』なら、TPJは『居場所の地図』」

### TPJではないもの・TPJであるもの

**❌ TPJではないもの**
- 料理の味・クオリティの評価
- 価格・コスパの比較ランキング
- 「一番うまい店」を決める権威
- 広告掲載・有料露出の媒体
- 飲食店だけを対象にした評価
- ユーザー口コミの集計サービス
- インスタ映え・バズりを評価する場

**✅ TPJであるもの**
- 「いる体験・居場所としての価値」の認証
- 業種横断（カフェ・ホテル・スパ等）の評価
- AIに引用される信頼ある一次情報源
- 掲載枠の希少価値（デジタル不動産）
- インバウンド向け日英両言語の発信基盤
- 7つの空間軸による独自評価システム
- 「日本のサードプレイス」の世界標準定義者

### 誰も真似できない理由（参入障壁）

1. **「サードプレイス」という概念の先行定義権**
   食べログが「飲食店の点数」を定義したように、TPJが「サードプレイスの価値基準」を先に定義した者が市場を制する。後から参入した者は「TPJに似ている」と言われ続ける。定義者は一社のみ。

2. **業種横断という構造的優位**
   食べログは飲食店しか評価できない。ミシュランは高級店中心。カフェ・ホテル・スパ・コワーキングを「サードプレイス」という一軸で横断評価できるのはTPJだけ。市場規模が20〜30兆円になる根拠もここにある。

3. **AIに引用される構造を最初に作る者が勝つ**
   GBC記事サイトはすでにSchema実装済み。AIが「おすすめのサードプレイスは？」に答えるとき、最初に蓄積されたデータソースを引用する傾向がある。先行者が圧倒的に有利なゲーム。

4. **「評価される側」が評価を望む構造**
   TPJの認証は「掲載されること自体がステータス」になる設計。認証を取りたい店舗がTPJに価値を提供するポジティブな循環が生まれる。

5. **インバウンド×日英両言語の空白地帯**
   「Japan's best third places」を英語で検索しても、権威ある情報源が存在しない。TPJがその空白を埋める唯一の候補。

### ポジショニング・ステートメント

> 食べログは「どこで食べるか」を決める。ミシュランは「どこが最高か」を決める。**TPJは「どこにいるか」を決める。**
>
> AI時代に人々が最も問うのは「おいしい店はどこか」ではなく、「今日、どこで自分らしくいられるか」だ。その答えを出せる権威は、世界にまだない。TPJがその権威になる。

## ファイル構成

- `project-docs/` : Web ページの内容や、その他作成に必要なドキュメントを格納
- `public/` : Cloudflare Pages で公開設定をするディレクトリ。ここに成果物を入れる
- `public/assets/` : 画像、CSS、JavaScript などの生成物の置き場

# 作成のためのルール

## 保存先

- ファイルは必ず **UTF-8** で保存（日本語の文字化け防止）
- 成果物は `public/` ディレクトリに保存
- 特に指示がなければ `public/index.html` で作成
- 指示があれば、その指示に合わせて `public/` 配下に保存
- 画像・CSS・JavaScript を生成した場合は `public/assets/` 配下の所定フォルダに保存

### ⚠️ Git操作の重要ルール（絶対厳守）

**🚨 絶対に守ること 🚨**

1. **ユーザーの明確な指示があるまで、コミットもプッシュも絶対に行わない**
2. **コミットの指示があった場合** → コミットのみ実行し、完了を報告。プッシュはしない
3. **プッシュの指示があった場合** → その時初めてプッシュを実行
4. **「コミット＆プッシュ」の指示があった場合のみ** → 両方を実行

**禁止事項:**
- ❌ 勝手にコミット＆プッシュを連続実行する
- ❌ 「コミットしました」と言って勝手にプッシュもする
- ❌ ユーザーの確認なしに任意のGit操作を行う
- ❌ 作業完了後に自動でコミット＆プッシュする

**このルールは何度も破られているため、各作業の完了時に必ず「コミット＆プッシュの指示をお待ちしております」と伝え、ユーザーの指示を待つこと。**

## ワンページ Web サイトの生成方法

ワンページサイトの作成・編集を行う際は、必ず **`one-page-site-builder` スキル**を使用すること。
このスキルに、技術スタック（Tailwind CSS v4 / Animate.css / AOS / Lucide）、HTML 構造、固定ナビゲーション仕様、セクション構成、画像最適化方針などの能力がまとまっている。

画像が必要な場合は **`unsplash-image-finder` スキル**を使って Unsplash から取得する。
