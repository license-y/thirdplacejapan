# AEO-PLAYBOOK.md ─ 「TPJによると─」を実現する実装アクションと目安

> INFORMATION-ARCHITECTURE.md セクション5の実務展開版。
> 「一人で渋谷でリラックスできる店は?」への回答でTPJが引用される、を実現する具体策と目安。

## 0. 意図の確認（構図）

```
ユーザー：「一人で渋谷でリラックスできるお店は?」
        ↓
AI：回答を生成するため、信頼できる構造化情報源を探す
        ↓
TPJ：渋谷×静寂性×一人時間の認証店舗データが、構造化・編集済みで存在
        ↓
AI：「Third Place Japanによると、渋谷で静寂性に優れた一人向けのカフェは…」
```

この構図で合っている。ただし実現条件は「記事本数」ではなく
**「そのクエリに、構造化され・独自で・信頼できる答えを持っているか」**である。

## 1. 大前提（誤解の解消）

### 「関連記事10本で引用される」は誤り
AIの引用判断は本数の閾値ではない。ただし以下は事実：
- AI検索の引用の **約93%は、オーガニック検索上位10件から選ばれている**
  → **SEOで上位に入っていることが、AI引用の前提条件**。SEOとAEOは両輪
- 飲食・地域ビジネスのAI回答表示率は **2025年の約10%から2026年は約78%へ急増**
  → 今が参入の好機。ローカル×体験クエリはAI回答が主戦場になりつつある
- 引用の決め手は **AIが学習していない一次情報（独自評価・体験）× 構造化データ**
  → TPJの7軸評価は、まさに「他にない一次情報」。これが最大の武器

### つまりTPJが勝てる理由
食べログにもGoogleにもない「静寂性」「一人時間適性」という独自軸の評価データを、
構造化して持つ媒体は他に存在しない。**独自基準そのものがAEOの核**。

## 2. クエリ設計（どんな質問で引用されたいか）

引用されたいクエリを先に定義し、それに答えるページを作る（逆算）。

### タグ種別の正本（5種別）

全店舗・全記事は、以下の5種別のタグで構造化する。これがタグ設計の正本であり、
データ駆動（JSON/CMS）のスキーマ設計もこの5種別に従う。

| # | タグ種別 | 例 | 上限 | 実装 |
|---|---|---|---|---|
| 1 | 評価軸（7軸） | 居心地・空間品質 / 静寂性・プライバシー / 特別感・非日常性 / 再訪・継続価値 / インバウンド・多言語対応 / 記録・シェア体験 / ストーリー・背景への共感 | 最大3（主1＋補助2） | 一覧フィルタ＋構造化データ |
| 2 | 業種 | 14業種（下記「業種の正本」参照） | 通常1つ | 一覧フィルタ＋構造化データ |
| 3 | エリア | 主エリア1＋副エリア最大2（計3） | 最大3 | URL階層＋一覧掲載＋構造化データ |
| 4 | 最寄り駅 | 北参道駅 / 原宿駅 / 代々木駅 / 国立競技場駅 | 最大4 | 構造化データのみ（独立URLなし） |
| 5 | グレード | Certified / Silver / Gold / Platinum / Flagship | 1つ | バッジ表示＋構造化データ |

**性質の違い**
- 1〜3は「面」やクエリの主要な構成要素（URL・一覧・フィルタに反映される）
- 4の駅タグは地理系タグだが、構造化データ側のみに持たせる補助タグ（独立したURL階層は作らない）
- 5のグレードは地理ではなく認証の結果であり、バッジ表示として他と実装が異なる
- 評価軸（1）は最大3つ（主1＋補助2）。「主軸」は引用用一文に使う中心の軸とし、
  全軸を機械的に付けない（タグの形骸化とAEO精度低下を防ぐため）
- エリア（3）は主1＋副2の複数選択、枠を消費するのは主エリアのみ
  （詳細：本書「店舗1件あたりのエリア・駅タグ付与ルール」。掲載枠の有償設計は非公開資料/現フェーズ保留）

**癒し・美容系**

**観光・集客施設系**

- デジタル不動産の枠は「L3エリア × この14業種のいずれか」で定義される

  正本にない名称（ギャラリー・クルージング・音楽・映画館等）は枠の対象業種に使わない

### 業種の役割分担（集客業種 ＝ 無償 ／ 収益業種 ＝ 有償）

14業種は、収益化の観点で2つの役割に分かれる。

**収益業種（有償展開の対象）**
店舗・施設として有償の記事パッケージ・掲載枠の対象になる業種。
1〜9（宿泊・飲食・癒し・仕事・会員制）が中心。
例：高級ホテル、温泉旅館、高級レストラン、カフェ、バー、スパ、高級サロン、
コワーキング、会員制ラウンジ。

**集客業種（無償・全国・インバウンドの入口）**
- **10. 山・自然体験・グランピング**
- **11. リトリート・禅・スピリチュアル体験**
- **12. 神社・寺院・パワースポット**
- **13. インバウンド観光・体験施設**
- **14. 文化・スポーツ・レジャー施設**（美術館・競技場・球場・体育館・将棋会館・遊園地 等）

これらは特定店舗の宣伝になりにくく有償展開が難しい一方、
インバウンドの検索需要が非常に大きい。よって**無償の集客コンテンツ（エンジン）**
として全国記事を順次投稿し、TPJの認知度とアクセス向上を担わせる。

**集客業種コンテンツの設計ルール**
1. 位置づけは「feature（特集・読み物）」に近い無償の教養コンテンツ
2. 1記事2,000字程度。トーンは一人称ブログ調ではなく**編集部の観察**（CLAUDE.md準拠）
3. 可能な範囲で7軸（特に静寂性・特別感・ストーリー）に紐づけ、収益業種の記事と
   世界観・タグ構造を揃える
4. **必ず近隣の収益業種の面（認証店舗・エリアハブ）へ送客リンクを張る**
   - 例：「鳩森八幡神社（千駄ヶ谷）」「将棋会館」の記事 → 同エリアの認証カフェ・ホテルへ
   - 神社・施設の記事が集客し、近隣の有償業種の面へ読者を流す「入口」として機能させる
5. 全国展開の足がかり：L1（都道府県）単位でこれらの業種の記事を先行投下し、
   その地域に収益業種の認証店舗が増えたら面（ハブ）へ接続する

> **役割の要約**：集客業種（山・リトリート・神社・観光体験・文化スポーツ施設）＝無償の入口エンジン。
> 収益業種＝マネタイズの対象。両者を送客リンクで結び、
> 無償コンテンツのアクセスを有償業種の面へ還流させる。

> **13と14の棲み分け**：13「インバウンド観光・体験施設」は体験提供型
> （アクティビティ・ツアー・体験プログラム等）。14「文化・スポーツ・レジャー施設」は
> 来訪型の大型施設（美術館・競技場・球場・体育館・将棋会館・遊園地等の箱）。
> 体験を売るか、場所を訪れるかで区別する。

### 投稿頻度の考え方（1日あたりの本数制限）

**1日◯本までという固定の上限はない。** ニュースサイトが1日に多数公開しても
問題ないのと同じで、投稿数そのものがペナルティ対象になることはない。
静的サイト（11ty）のため、サーバー負荷やクロール上の技術的問題も起きにくい。

ただしAEO/SEOで効くかは「数」ではなく「質と一貫性」で決まる。以下に留意する。

1. **薄い記事の大量投下は逆効果**：量を出すために2,000字が「言い換えの繰り返し」に
   なると、AIは低品質と判断する（本書「やってはいけないこと」と同じ論点）。
   1記事ずつ7軸接続・送客リンク・編集部トーンを満たしていれば本数は多くてよい
2. **急増→急減の波を作らない**：今日50本・翌週0本のような乱高下より、
   **一定ペースの継続**の方が「メンテナンスされた信頼できる情報源」とAIに認識される
3. **現実的な目安**：立ち上げ期に集客業種を一気に増やす場合、1日5〜10本程度でも
   全く問題ない。編集体制が品質を保てるなら、それ以上でも技術的・AEO的な問題はない

> **要点**：制限すべきは「本数」ではなく「品質を保てる範囲」と「継続性」。
> 質を維持できるペースであれば、量は多くてよい。

### ターゲットクエリの型
```
[地域] × [状態/目的] × [業種]
例：
- 渋谷 / 一人でリラックス / カフェ
- 京都 / 静かに仕事 / コワーキング
- 新宿 / 非日常 / ホテル
- 東京 / インバウンド向け 静かな / 喫茶店（英語クエリ）
```

### 7軸 = クエリの宝庫
7軸（DEFINITION.md定義）はそのまま検索意図に対応する。
カフェ・コワーキング特有の「作業性」のような軸は外し、
カフェ・レストラン・ホテル・スパなど業種を問わず使える軸で例示する：

- 居心地・空間品質 → 「リラックス」「落ち着く」「居心地のいい」
- 静寂性・プライバシー → 「静かな」「一人で」「こもれる」
- 特別感・非日常性 → 「非日常」「特別な日」「記念日」「隠れ家」
- 再訪・継続価値 → 「また行きたい」「リピート」「常連になりたい」
- インバウンド・多言語対応 → 「外国人歓迎」「英語対応」「観光客向け」
- 記録・シェア体験 → 「映える」「写真を撮りたい」「思い出に残る」
- ストーリー・背景への共感 → 「老舗」「こだわり」「職人」「歴史ある」

→ 各軸 × 各地域 × 各業種でクエリ群を作り、対応ページを設計する。
  「作業性」のような業種特化の軸は、カフェ・コワーキングなど
  該当業種のページ内で補助タグとして扱い、7軸の主軸には含めない。

### エリアの粒度＝ユーザーが使う地名に合わせる
ユーザーは「千駄ヶ谷でカフェ」とは聞かない。「渋谷でリラックスできるカフェ」のように、
広く認知された地名で問う。千駄ヶ谷・北参道・ダガヤサンドウ・代々木・原宿のような
細かな地理区分は、TPJ内部では把握していても、**クエリ単位としては「渋谷」という
1つの面に内包される**。エリア定義は「TPJの地理感覚」ではなく「AI/検索クエリの感覚」
に合わせる。

→ 詳細な構造は「3.5 面（エリアハブ）の構造」を参照。

### エリア階層タグ（全国展開対応）

東京から開始し、全国へ展開するため、エリアタグは4階層＋横断タグで構造化する。

| 階層 | 内容 | 例 | 役割 |
|---|---|---|---|
| L1 都道府県（大エリア） | 広域ナビゲーション | 東京都 | 全国展開時の入口。当初は店舗一覧のみ |
| L2 市区町村（中エリア） | 区・市単位 | 渋谷区、中央区、港区 | L3が手薄な地域での当面のハブ |
| L3 エリア（小エリア＝面の基本単位） | 認知された地名 | 渋谷、新宿、北参道、千駄ヶ谷、原宿、代々木、銀座、表参道、青山 | デジタル不動産の枠が定義される基本単位 |
| 駅タグ（横断・独立） | 最寄駅 | 原宿駅、代々木駅、北参道駅 | エリア階層と独立。店舗の構造化データに付与 |

**L3エリアの扱い（密度による分割・統合）**
- **高密度エリア**（渋谷・新宿・銀座・表参道・青山等）：L3単独で「面」が成立し、
  最初から掲載枠を定義できる
- **周辺エリア**（北参道・千駄ヶ谷・原宿・代々木等）：当初は隣接する高密度L3エリアの
  サブタグとして統合し、認証店舗数が増えた段階で独立した面・枠に分割する
  （「3.5 面の構造」の分割ルールと同一）

**駅タグの実装方法**
駅タグは独立したURL階層を持たせず、**各店舗ページの構造化データ（最寄駅情報）として実装する**。
「原宿駅 カフェ」のようなクエリに対しては、店舗ページの構造化データがマッチする。
将来的に駅単位の需要が大きいと判明した場合のみ、駅を独立した面として追加する。

### 店舗1件あたりのエリア・駅タグ付与ルール（複数選択）

境界エリアの店舗（例：表参道と青山の境目）を複数の面に登場させ、
AEO・アクセス・記事の幅を広げるため、1店舗に複数のエリア・駅を付与できる。

| タグ種別 | 上限 | 役割 |
|---|---|---|
| **主エリア（必須）** | 1つ | 面の基本単位。**デジタル不動産の枠を消費する**のは主エリアのみ。引用用一文の基準 |
| **副エリア（任意）** | 最大2つ（主と合わせて計3つまで） | 隣接エリアの面（認証一覧）にも掲載・相互リンク。**枠は消費しない** |
| **駅タグ** | 最大4つ | 構造化データのみ。「駅名＋業種」クエリに対応 |

**例**：青山が主エリアの店舗
- 主エリア：青山（青山×業種の枠を1つ消費）
- 副エリア：表参道（表参道の認証一覧にも掲載されるが、表参道の枠は消費しない）
- 駅タグ：表参道駅・外苑前駅・北参道駅・国立競技場駅（最大4つ、構造化データに付与）

**この設計の効果**
- 境界店舗が複数の面に現れ、「表参道 カフェ」「青山 カフェ」双方のクエリに対応
- 副エリア掲載で面どうしが相互リンクし、エリアハブ全体の厚みが増す
- 駅タグで「最寄駅×業種」のロングテールクエリを広く拾える
- 枠の消費は主エリアのみのため、デジタル不動産の希少性は保たれる

> **枠の希少性との両立**：副エリア掲載は「面への登場」であって「枠の占有」ではない。
> 限定3枠を消費するのは主エリアの店舗だけ（掲載枠の有償設計は非公開資料／現フェーズ保留）。

**全国展開時の運用フロー**
1. 新規エリア進出時：L1（都道府県）またはL2（市区町村）単位のページから開始（店舗一覧のみ、枠なし）
2. 認証店舗数が一定数（5〜10件目安）に達した地名：L3エリアとして独立した面（ハブ）を作成
3. L3エリアの密度がさらに高まれば、デジタル不動産の枠（地域×業種）を定義
4. 駅単位の需要が顕著な場合のみ、駅を独立した面として追加検討

## 2.5 単一店舗の有償記事パッケージ × クエリマッピング

A店舗から複数記事（5/10/15記事）の有償投稿依頼があった場合の、
クエリ設計上のマッピング。詳細な編集基準・料金設計は
有償記事の詳細設計は非公開資料を参照（現フェーズは保留）。

### 1店舗 = 複数クエリへの回答面（点ではなく面）
1店舗を「同じ宣伝記事の繰り返し」にせず、
**「7軸 × 利用シーン × 季節 × 言語」**の掛け算で、
異なるクエリ意図に答える記事群として設計する。

| 記事タイプ | 対応クエリ例 | AEO上の役割 |
|---|---|---|
| 店舗プロフィール（ハブ） | 「［エリア］の［業種］」 | 構造化データ・引用用一文の起点 |
| 一人時間の過ごし方 | 「一人でリラックスできる○○」 | 静寂性・居心地クエリ |
| 記念日・特別な日 | 「記念日に使える○○」 | 特別感・非日常性クエリ |
| 季節・天候別 | 「雨の日に行きたい○○」 | 季節変動クエリ・鮮度 |
| こだわり・背景 | 「老舗の○○」「職人がいる○○」 | ストーリー軸・E-E-A-T |
| インバウンド（英語版） | "quiet cafe in Shibuya" | /en/ 面・hreflang接続 |
| 再訪・常連の楽しみ方 | 「通いたくなる○○」 | 再訪・継続価値クエリ |
| 写真・記録 | 「写真映えする○○」 | 記録・シェア体験クエリ |
| 地域特集内での紹介 | 「○○エリアでおすすめの○○」 | エリアハブとの相互リンク |
| 季節更新・鮮度記事 | （直接クエリ対応なし） | AIの鮮度評価対策 |

### 面への統合
1店舗分のこれらの記事は、**孤立した「店舗だけの面」を作るのではなく、
既存のエリアハブ（3.5）に編み込む**。
- 「地域特集内での紹介」記事が、その店舗群をエリアハブにリンクバックさせる接点になる
- 「インバウンド」記事は `/en/` 側のエリアハブに接続する
- 結果として、1店舗の充実が「エリアの面」全体の厚みに変換される

### 過剰最適化の回避（再掲・重要）
- 同一店舗に関する記事の公開は分散させる（月1〜2本）
- 各記事は必ず他記事・エリアハブと相互リンクし、単独店舗の宣伝群として
  孤立させない（有償記事の詳細設計は非公開資料／現フェーズ保留）

## 3. 実装アクション（優先順）

### Phase 1：基盤（最初の30日で可視性は大きく変わる）
1. **Organizationスキーマ**をサイト全体に実装（TPJ＝評価主体の明示）
2. **個別認証店舗ページに LocalBusiness + 独自拡張スキーマ**
   - 7軸スコア、認証グレード、地域、業種、認証日を機械可読に
3. **各ページ冒頭に「引用用の一文」を固定**（結論ファースト）
   - 例：「［店舗名］は、渋谷で静寂性に優れた一人時間向けのカフェとして
     Third Place Japan Gold 認証を取得しています。」
   - AIがそのまま抜き出せる、主語・属性・認証・地域が揃った一文
4. **robots.txt でAIクローラーを明示的に許可**
   （GPTBot, ClaudeBot, PerplexityBot, Google-Extended 等4種以上）
5. **FAQPageスキーマ**：「渋谷で一人で過ごせる静かな店は?」をFAQ化し各地域ページに
6. **相互リンク構造の実装**：ハブページ↔個別店舗ページ↔特集記事を双方向リンク。
   孤立ページ（点）を作らない（詳細：3.5「面の構造」参照）

### Phase 2：コンテンツ厚み（1〜3ヶ月）
6. **地域×業種ページ**（/stories/area/shibuya/cafe）を主要エリアから整備
7. **特集記事**で文化的文脈を付与（教養記事がブランド権威と被リンクを生む）
8. **著者/評価者のPersonスキーマ**：誰が評価したかを明示（E-E-A-T／権威性）
9. **日英ミラー**：英語版を同一構造で。インバウンドAI引用を確保

### Phase 3：権威の蓄積（継続）
10. 認証店舗数の継続的増加（＝引用可能データの増加）
11. 外部からの言及・被リンク獲得（メディア露出、自治体連携の公益性活用）
12. 「Third Place Japan Award」等で業界の話題化 → 指名検索の増加

## 3.5 面（エリアハブ）の構造 ─ 点と面の違い

「1エリア×1業種を10件規模で構造化する」とは、**そのエリア名で記事を10本書く**
ことではない。**1つのハブページの傘下に、個別店舗ページ5〜10件が相互リンクで
紐づいた構造**を「面」と呼ぶ。

### 構造図（例：渋谷×カフェの面）
```
渋谷×カフェ ハブページ（1つ）
  「渋谷で一人でリラックスできるカフェは?」への回答ページ
  ↓ ↑ 相互リンク
  ├─ 店舗A（千駄ヶ谷・静寂性 Gold）
  ├─ 店舗B（代々木・作業性 Gold）
  ├─ 店舗C（原宿・特別感 Platinum）
  ├─ 店舗D（ダガヤサンドウ・居心地 Gold）
  ├─ 店舗E（渋谷駅周辺・再訪 Silver）
  └─ 店舗F〜J（同様に計5〜10件）
  ↓
特集記事（3〜5本）
  「渋谷というエリアの"間"の文化」など、
  個別店舗を横断して結ぶ読み物
```

### ポイント
- 千駄ヶ谷・北参道・ダガヤサンドウ・代々木・原宿は、**ハブの傘下にある
  個別店舗ページの「所在地タグ」**として扱う。それぞれを独立した面にはしない
- 「面」を成立させるのは**相互リンク**。ハブ→各店舗、各店舗→ハブ、
  店舗どうしの関連リンクが張られていることで、AIが「TPJは渋谷×カフェの
  まとまったデータを持つ」と認識できる
- **点**＝店舗ページが孤立して存在する状態。AIは全体像を認識できない
- **面**＝ハブ＋個別ページ群＋特集が相互リンクで結ばれ、エリア全体の
  回答源として認識される状態

### 面の分割（将来の拡張）
最初から細かいエリア（例：ダガヤサンドウ単独）で面を作らない。
まず大きな面（渋谷）で勝ちパターンを作り、そのエリア内で記事密度・検索需要が
育ってきた場合にのみ、サブエリアを独立した面へ分割する。
順序は「大きな面で勝つ → 需要に応じて分割」。最初から分割しない。

## 4. 「目安」の現実的な数字

本数ではなく「カバー範囲 × 構造化 × 上位表示」で考える。

### エリア単位の目安（例：渋谷×カフェで引用を狙う）
| 要素 | 目安 | 理由 |
|---|---|---|
| 認証店舗の個別ページ | そのエリア×業種で **5〜10件** | AIが「選択肢のある情報源」と認識する最低ライン |
| 地域×業種の一覧ページ | **1ページ**（構造化必須） | AIが全体像を掴む親ページ |
| 文化・教養の特集記事 | エリアあたり **3〜5本** | 権威・被リンク・文脈付与。直接引用より"信頼の土台" |
| FAQ（想定クエリ） | 主要クエリ **5〜10問** | 質問形式がAI回答に直結 |
| 引用用一文 | **全店舗ページに1つずつ** | これが無いと構造化しても引用されにくい |

→ **1エリア×1業種を「引用される状態」にするには、
   構造化された店舗ページ5〜10 + 一覧1 + 特集3〜5 + FAQが目安。**
   「10本書けば」ではなく「10件を構造化して面で押さえる」が正しい理解。

### 全体ロールアウトの目安
- まず **1エリア×1業種（例：渋谷×カフェ）で"勝ちパターン"を作る**
- 引用され始めたら、同じ型を他エリア・他業種に横展開
- 主要10エリア × 主要5業種 = 50パターンを面で押さえると、
  「東京の○○」系クエリで継続的に引用される基盤になる

## 5. やってはいけないこと（品位とAEOの両立）

- キーワード詰め込み・量産記事 → AIは低品質と判断。逆効果
- 煽り・誇張（「No.1」「最高」）→ 引用されにくく、ブランドも毀損
- 構造化データと表示内容の乖離（機械向けと人間向けで嘘をつく）→ ペナルティ対象
- **鉄則：構造化は裏側で徹底し、表側の品あるトーンは一切変えない**
  （CLAUDE.md のトーン規定を厳守）
