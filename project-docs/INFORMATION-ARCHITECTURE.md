## 0. 全体アーキテクチャ（2サイト + 1物理拠点）

```
┌─────────────────────────────────────────────────┐
│  Third Place Japan エコシステム                    │
├─────────────────────────────────────────────────┤
│                                                 │
│  ① メインサイト（法人向け / 思想・制度）            │
│     thirdplacejapan.com                         │
│         │                                       │
│         │  認証された場所を「読む」へ送客           │
│         ▼                                       │
│  ② 記事サイト（個人向け / 文化・店舗・日英）         │
│     thirdplacejapan.com/journal （サブディレクトリ）│
│         │                                       │
│         │  文化的興味 → 認証制度の理解へ還流        │
│         ▲                                       │
│         └───────────────────────────────────┘   │
│                                                 │
│  ③ GBC（物理拠点 / 思想の実証）                    │
│     メイン・記事の両方から参照される共通アンカー      │
└─────────────────────────────────────────────────┘
```

### ドメイン方針
- **同一ドメイン・サブディレクトリ構成**を採用（`/journal`）
  → SEO/AEOの権威を1ドメインに集約。被リンク・引用評価を分散させない
- 英語版は `/en/`（記事サイト配下に多言語ディレクトリ）
  → `thirdplacejapan.com/en/journal/...`
- サブドメイン（journal.〜）は採らない。AEO上、評価分散を避けるため

## 1. メインサイト IA（法人向け）

### グローバルナビ（最小限・5項目以内）
```
[ TPJについて ]  [ 評価の基準 ]  [ 認証について ]  [ Journal ]  [ お問い合わせ ]
```
- ロゴは常に左上、トップへ還る
- 「For Business」「料金」はナビに置かない（フッターのみ）

## 2. 記事サイト IA（個人向け・日英）

### グローバルナビ
```
[ 読み物 ]  [ 場所をめぐる ]  [ 認証店舗 ]  [ English ]  [ TPJとは ]
```
- 「TPJとは」はメインサイトへの還流口
- ナビに広告的要素・CTAを置かない

### ページ階層（URL）─ 全国展開対応の4階層

エリアは「都道府県 > 市区町村 > エリア(面)」の階層で表現する。
駅タグは独立階層を持たず、店舗ページの構造化データで表現する
（詳細：AEO-PLAYBOOK.md「エリア階層タグ」参照）。

```
/journal                                          記事トップ（特集・新着）
/journal/feature/{slug}                           特集・読み物（教養・文化・思想）
/journal/story/{slug}                             店舗ストーリー（有償記事もここに同化）

/journal/area/{prefecture}                        L1 都道府県インデックス
                                                   （新規進出エリア。店舗一覧のみ、枠なし）
/journal/area/{prefecture}/{city}                 L2 市区町村インデックス
                                                   （L3が手薄な地域の当面のハブ）
/journal/area/{prefecture}/{city}/{area}          L3 エリア（面）一覧
/journal/area/{prefecture}/{city}/{area}/{category}
                                                   L3 エリア×業種ハブ（掲載枠が並ぶ層）
                                                   ※{category}＝14業種（正本：AEO-PLAYBOOK.md「業種の正本」）

/journal/certified                                認証店舗一覧（グレード別・地域別）
/journal/certified/{venue-slug}                   個別認証店舗ページ（AEOの中核・後述）
/en/journal/...                                   英語版（同構造のミラー再編集）
```

**例（東京・渋谷・カフェの場合）**
```
/journal/area/tokyo                  東京都（進出初期のインデックス）
/journal/area/tokyo/shibuya-ku       渋谷区（L3が手薄な周辺エリアの統合先）
/journal/area/tokyo/shibuya-ku/shibuya          渋谷エリア（面）一覧
/journal/area/tokyo/shibuya-ku/shibuya/cafe      渋谷×カフェ（掲載枠が並ぶハブ）
```

**段階的成長のルール**
- 新規都道府県進出時：L1ページのみ（店舗一覧）からスタート
- 認証店舗数が増えた地名：L3エリアとして面（ハブ）を作成
- L3の密度が十分になれば、L3×業種に掲載枠を定義
- 周辺の小エリア（北参道・千駄ヶ谷等）は、当初は隣接するL3（渋谷）に統合し、
  将来的に独立したL3として分割（AEO-PLAYBOOK.md「面の分割」参照）

### コンテンツ分類（CLAUDE-v2.md セクション4に対応）
- feature＝無償の教養記事（シェア・SEO/AEO資産の中核）
- story＝店舗記事（無償・有償が品質上見分けつかない形で混在）
- area＝地域ガイド＋掲載枠（EDITORIAL-COMMERCE.md の見せ方に従う）
- certified＝認証の信頼性可視化＋AI引用の対象データ

### フッター（記事）
```
読み物 / 場所をめぐる / 認証店舗 / English
─────
TPJの認証制度について（→ メインサイト）/ 運営会社 / お問い合わせ
─────
© Third Place Japan
```

## 3. 回遊導線（2サイト間の往復設計）

### メイン → 記事（思想を持った人を、文化へ）
- トップのGBC／認証の意味セクション末から「認証された場所を読む → Journal」
- ナビ常設の [ Journal ]
- 押し付けず、"続きを読む"の自然な延長として配置

### 記事 → メイン（文化に触れた人を、制度の理解へ）
- 個別認証店舗ページ末に「この場所は Third Place Japan 認証店舗です → 認証制度を知る」
- ナビ常設の [ TPJとは ]
- 店舗オーナーの目に留まる位置（store storyの末尾）に控えめな For Business 動線

### 重複の解消
- **認証店舗一覧の正は「記事サイト `/journal/certified`」**に一本化
- メインサイトは制度の"説明"のみを担い、店舗の"一覧・実体"は持たない
- メインの「認証の意味」セクションからは記事サイトの一覧へリンクするのみ

## 4. GBC の位置づけ

- **独立ドメインにはしない。** メインサイト配下 `/certification/gbc` を正とする
- 理由：GBCは「認証制度の最上位（Flagship）を体現する実証例」であり、
  制度の文脈（＝メインサイト）の中にあってこそ意味を持つため
- 記事サイトからは `story` として文化的に紹介し、詳細は `/certification/gbc` へ送る
- 二重管理を避け、GBCの"制度的意味"はメイン、"体験の物語"は記事、と役割分担

## 5. AEO実装（事業の核心を構造に落とす）

「TPJによると─」がAIに引用される構造を、設計レベルで担保する。

### 引用される単位＝個別認証店舗ページ（/journal/certified/{venue-slug}）
このページを「AIが引用しやすい構造」で設計する：

1. **構造化データ（schema.org）を全店舗ページに実装**
   - `LocalBusiness` + 独自拡張で7軸スコア・認証グレードを明示
   - 認証日・グレード・地域・業種を機械可読に
2. **一文で引用できる要約を各ページ冒頭に固定配置**
   - 例：「［店舗名］は、東京・渋谷の静寂性に優れたカフェとして
     Third Place Japan Gold 認証を取得しています。」
   - AIがそのまま引用できる、主語・属性・認証が揃った一文
3. **認証データの一覧をJSON-LD／オープンな形式で公開**
   - `/journal/certified` を構造化し、AIクローラが全体像を取得可能に
4. **日英両方を同一構造で**：英語圏AI・インバウンド検索の引用元も確保
5. **権威の明示**：各ページにTPJ独自基準である旨と評価主体を明記し、
   引用時に「TPJによると」が自然に付く文脈をつくる

### AEOの鉄則
- 構造化データは裏側（機械向け）。表側の品位あるトーンは一切変えない
- AEO目的の煽り・キーワード詰め込みはしない。質の高い編集こそ最大のAEO

## 6. 共通基盤

### 技術スタック（確定）
- **メインサイト（法人向け）：静的1ページ（素のHTML/CSS/JS）**
  - VSコードで直接記述。フレームワーク不要
  - 理由：更新頻度が低く構成が固定的。軽量・高速・品ある佇まいと相性が良い
  - 構造化データ（JSON-LD）は各セクションに直接埋め込む
- **記事サイト（個人向け・日英）：11ty（Eleventy / 静的サイトジェネレータ）**
  - 理由：完全な静的HTML出力でAIクローラーが確実に読める（AEOの核心と直結）
  - 多言語（`/en/`・hreflang）、テンプレートによるJSON-LD自動展開、高速表示に対応
- **店舗データの管理：データ駆動を原則とする**
  - 7軸スコア・グレード・地域・業種・引用用一文を JSON/CSV または
    ヘッドレスCMS（microCMS等）で一元管理し、11tyがページを自動生成する
  - 理由：店舗増加に耐える保守性。構造化データと引用用一文を
    データから機械的・一貫して生成でき、AEO品質を均一に保てる
  - 手書きMarkdownでの個別記事化は、特集・読み物（feature）に限定する

### URL・技術方針
- 1ドメイン集約、HTTPS、クリーンURL（slug）
- 多言語は `/en/` ディレクトリ、hreflang を相互設定
- パンくず：記事サイトの階層ページに設置（トップ > 場所をめぐる > 渋谷 > カフェ）
  メインサイトは浅い階層のため不要

### サイトマップ（XML / 検索・AI向け）
- 全公開ページをXMLサイトマップ化し、更新を継続的に通知
- 認証店舗ページの追加が、そのまま引用可能データの増加になる設計
